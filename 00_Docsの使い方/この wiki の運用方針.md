# この wiki の運用方針

## 目的

この wiki の目的は、Git / Codex / repo 管理の内部構造を完全理解することではない。  
自分の認知補助・復習・整理・外部化のために、短期間で更新可能な仕組みを作ることを優先する。

この repo 自体の完成度を目的にしすぎず、
使いながら安定させ、必要な知識は後から足していく方針で運用する。

## 基本姿勢

- 内部構造の完全理解より、運用の安定を優先する
- 0から仕組みを全部理解してから使うのではなく、使いながら必要な範囲だけ理解する
- 確実性を上げるために時間をかけすぎるより、短期間で回る状態を作る
- 壊れる可能性はある前提で進める
- 壊れたら終わりではなく、止まって見直せるようにする

## この wiki の役割

この wiki は、知識をきれいに保存するだけの場所ではない。  
以下のための外部化装置として使う。

- 自分が何を考えていたかを後で拾い直す
- 途中の判断や仮ルールを残す
- 復習のための整理された入口を作る
- ChatGPT / Codex に毎回同じ前提を渡しやすくする
- 更新を続けながら構造を安定させる

## 深く知らなくてもよいもの

以下は、最初から深く理解していなくてもよい。

- Git の内部実装
- Codex の内部処理の詳細
- ブランチや sandbox の細かい理論
- repo 管理の仕組み全体の完全把握

これらは必要になった時に個別に補えばよい。

## 最低限見ておくもの

一方で、運用のために最低限確認すべきものはある。

- 今どこを触っているか
- どのファイルを正とするか
- どの変更が入るか
- いま進めて安全か
- 止まるべき条件は何か

完全理解よりも、この確認点を外さないことを重視する。

## この運用で大事にすること

### 1. 完全理解より、回る状態を作る
まずは使える形にする。  
理解は、使う中で必要になったところから補う。

### 2. 壊れても止まれるようにする
適当に流すのではなく、壊れた時に止まって確認できるようにする。  
そのために README / AGENTS / CURRENT などの固定点を持つ。

### 3. 自然言語だけに頼りすぎない
自然言語の指示は便利だが、少しずつズレることがある。  
そのため、継続して使う判断軸は repo 内の文書に固定する。

### 4. 認知負荷を下げる
毎回同じことを思い出すコストを減らす。  
構造・ルール・現在地を外に出し、頭の中だけで保持しない。

## リスクの考え方

この運用では、ある程度のリスクは許容する。  
素人の自分が、すべてを確実に理解してから進めるのは現実的でない。

その代わりに、

- source of truth を決める
- 確認ポイントを固定する
- 危ない時は止まる
- 更新しながら安定させる

という方針で進める。

## 結論

この wiki は、仕組みを完璧に理解した人だけが使うものではない。  
内部構造の完全理解よりも、

- 認知補助として使えること
- 復習に使えること
- 更新を続けられること
- 壊れても止まって見直せること

を優先する。

以後の運用では、
「深く知ってから使う」ではなく、
「最低限の確認点を持って使いながら安定させる」を基本方針とする。
