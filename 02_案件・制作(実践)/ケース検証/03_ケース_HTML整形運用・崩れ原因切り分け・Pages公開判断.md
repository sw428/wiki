# 03_ケース_HTML整形運用・崩れ原因切り分け・Pages公開判断

## 1. 事実（案件文脈）

- 対象は、ポートフォリオ実装中に発生した混在ログ（HTML整形設定 / 崩れ原因調査 / GitHub Pages公開運用 / その他話題）。
- このケースでは、フロントエンド実務に直接関係する論点だけを抽出する。
  - HTML整形（Prettier と VS Code標準フォーマッターの分担）
  - `/>` 置換の安全範囲
  - `.portfolio-page` の崩れ原因切り分け（SCSS断片あり）
  - GitHub Pages の公開元ブランチ確認
- 健康・私生活系の話題は、repo方針に従って記録対象外にした。

## 2. 判断プロセス

### ケース1: HTML整形の運用をどう分担するか

#### 候補A: HTMLも含めてPrettierで統一

- 判定: 条件付き不採用
- 理由: ルール統一はしやすいが、空要素が `/>` へ寄りやすく、今回の目的（`>` 寄り）と衝突しやすい。

#### 候補B: HTMLはVS Code標準、CSS/SCSS/JSはPrettier

- 判定: 採用（既定）
- 理由: HTML整形の意図（`>` 寄り）と、CSS/JS整形の安定性を両立できる。
- 実務メモ:
  - `bracketSpacing` は JS / JSON / JSX の `{}` 内スペース設定であり、HTMLの `/>` には影響しない。
  - `html.format.wrapAttributes` は `auto` から始め、必要時のみ `force-expand-multiline` に上げる。

#### 候補C: 保存時整形を切り、必要箇所だけ `Format Selection`

- 判定: 条件付き採用
- 理由: 「`<head>` を触りたくない」など局所要件がある時に有効。
- 例外: ファイル全体の整形一貫性は落ちるため、常用はしない。

### ケース2: `/>` を一斉置換してよいか

#### 候補A: 全ファイルで ` />` -> `>` を一括置換

- 判定: 不採用
- 理由: `.svg` やインラインSVGまで巻き込むと、XML系自己終了タグを壊すリスクがある。

#### 候補B: `*.html` に限定し、SVGを除外して置換

- 判定: 採用
- 理由: HTML空要素（`meta` / `img` / `input` / `br` など）だけを安全側で処理できる。
- 実務メモ:
  - HTML内に `<svg>...</svg>` がある場合は、置換前に目視確認する。
  - 可能なら置換よりフォーマッター運用で再発防止する。

### ケース3: `.portfolio-page` 崩れ原因の切り分け

#### 観測した論点

- PC用に `grid-column` / `grid-row` を個別指定したアイテムがある
  - 例: `.portfolio__item--dddone`, `.portfolio__item--work-media`
- `.portfolio__list:first-child { grid-column: 1 / -1; }` がある
- `aspect-ratio` はSP/PCで切り替えている
- 「`div` と `figure/figcaption` の違いで崩れたか？」という疑問が出た

#### 採用判断（主因の優先順）

1. **Grid配置指定の整合崩れを先に疑う**
   - クラス名不一致（例: `--dddone` の綴り）や、`grid-row` 固定による穴あきが起きると、見た目崩れに直結しやすい。
2. **`grid-column` の適用対象を確認する**
   - `grid-column` は grid item に効く指定。コンテナ側を選んでいると意図通り効かない。
3. **`div` と `figure` の違いは主因にしない**
   - ここは意味付けの差が中心。崩れは多くの場合、CSS（配置・サイズ・余白）側で起きる。
   - 例外として、`figure` の既定マージン未リセットは余白差の原因になり得る。

### ケース4: GitHub Pagesの公開先判断

#### 候補A: `dev_sub` に push すれば常に公開される前提で運用

- 判定: 不採用
- 理由: Pages の公開元が `main` のままなら、`dev_sub` push だけでは公開に反映されない。

#### 候補B: 公開元ブランチを先に確認してから運用

- 判定: 採用
- 理由: 「コード反映」と「公開反映」を混同しにくい。
- 確認順:
  1. `Settings -> Pages` で Source branch を確認
  2. `Actions` でデプロイ成否（緑チェック）を確認
  3. 公開URLを確認し、必要なら `Ctrl + F5` でキャッシュを外す

## 3. 案件固有として残すもの（02側）

- このログでは、`portfolio` / `main` / `dev_sub` という実運用前提で話が進んでいる。
- `.portfolio-page` の崩れ調査は、当該ページのクラス名・配置指定・カンプ要件に依存する。
- HTML整形の「どこまで縦に割るか（`auto` か `force-expand-multiline` か）」は、チームの可読性優先度に依存する。

## 4. 本線へ戻すもの（01/04側）

- 再利用可能な原理:
  - フォーマッター設定は「言語スコープ」と「目的（見た目 / 安全性）」で分担する。
  - 一括置換は対象拡張子を絞り、仕様の異なる文法（HTMLとSVG/XML）を混ぜない。
  - 崩れ調査は「意味タグ差」より先に「配置責務（grid/flex）」「適用対象」「計算結果」を確認する。
  - Pagesは「公開元ブランチ確認 -> デプロイ結果確認 -> 公開URL確認」の順で見る。
- 判断順:
  1. 何を固定したいか（見た目か運用か）を先に固定
  2. 設定の効く範囲（言語 / ファイル / 要素）を確認
  3. 危険な一括操作は対象限定 + 例外チェックで実行
  4. 公開運用は「push」と「deploy」を分離して検証
- 例外条件:
  - ファイル全体整形が副作用を生む場合は、`Format Selection` を使う。
  - インラインSVGを含むHTMLは、機械置換前に目視チェックを入れる。
