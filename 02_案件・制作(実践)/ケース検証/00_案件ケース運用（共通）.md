# 00_案件ケース運用（共通）

## このファイルの目的

- 案件ごとのケースバイケース判断を `02_案件・制作` 側で受けるための共通ルール
- そのケースから、再利用可能な原理だけを `01_学習本線` へ戻すための基準を固定する

## 基本ルール

- 案件で起きた判断は、まずこの配下でケース化する
- ポートフォリオの公開・記録範囲は `../ポートフォリオサイト/00_公開・記録方針.md` を参照する
- 案件側には指示内容の詳細ではなく、自分が意識した実装・判断理由・工夫を記録する
- 1ケース1ファイルを基本にする
- 同一回で議題が複数ある場合は、1ファイル内を `ケース1 / ケース2 / ...` で分割する
- 短い要約より、途中判断が追える説明を優先する
- ケース本文には「案件固有の事情」と「再利用可能な原理」を分けて書く

## 2つのケース入口を先に分ける

貼り付けログや対話ログを受けたら、反映先を先に決めず、まずケース検証の入口を分ける。

- 特定ポートフォリオ・特定案件で起きた判断  
  -> `02_案件・制作/ケース検証`
- 案件から離れた学習寄りの単独検証  
  -> `01_学習本線/ケース検証`

同じログに両方が混ざる場合は、1ファイルへ無理に統合せず、必要に応じてケースを2つに分ける。

- 例: `マリンプラクティスの2カラム/CTA/Voice配置`  
  -> 特定ポートフォリオのケースとして `02_案件・制作/ケース検証`
- 例: `BEM命名全般` / `SCSS分割運用`  
  -> 実務汎用ルールのケースとして `02_案件・制作/ケース検証` で受け、反映先は `02_案件・制作/BEM運用` / `02_案件・制作/SCSS導入`
- 例: `inline-flex` / `line-height` / `aspect-ratio` の基本原理  
  -> ケースで検証後、必要分だけ `01_学習本線(基礎概念)/HTML・CSS` へ戻す

## ケース後の反映先判定

- `01_学習本線` へ戻すのは、案件を超えて使える原理・判断軸・確認順だけ
- 特定ポートフォリオの判断・素材依存・その案件での実践記録は、対象案件フォルダへ残す
- BEM / SCSS のような実務汎用ルールは、学習本線へ自動移動せず、指定された実務ノート（例: `BEM運用` / `SCSS導入`）へ入れる
- 特定デザイン、特定クライアント事情、納期由来の妥協点は案件側に残す
- 本線側は日時ログとして追記せず、既存カテゴリに統合する
- 判断が割れる場合は、ケースファイル内の「反映先判定」に理由を残してから移す

## 記録テンプレート

```md
# xx_ケース_（タイトル）

## 1. 事実（案件文脈）
- 何を作っていたか
- どの制約があったか
- 何が問題として出たか

## 2. 判断プロセス
- 候補A:
- 候補B:
- 採用判断:
- 理由:

## 3. 案件固有として残すもの（02側）
- その案件でしか効かない条件
- 素材依存の調整値

## 4. 本線へ戻すもの（01側）
- 再利用可能な原理:
- 判断順:
- 例外条件:

## 5. 案件制作へ戻すもの（02側）
- 特定案件フォルダへ残すもの:
- BEM/SCSSなど実務汎用ノートへ残すもの:

## 6. 反映先判定
- 反映先:
- 判定理由:
- 反映内容（要約）:
```
