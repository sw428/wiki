# wiki

このリポジトリは、GPTとのリアルタイム対話で生まれたメモ・案件知識・学習内容・説明方針を、Codex が最終的に整理して統合するための wiki です。

> [!IMPORTANT]
> この repo は「知識を置くだけの倉庫」ではなく、再把握しやすく、判断を再利用しやすい外部脳として運用する。

> [!NOTE]
> 司令塔の核は ChatGPT 側に置き、Codex は整理・反映・修正の実作業担当として使う。

> [!TIP]
> 運用中に `AGENTS.md` / `CURRENT.md` / `README.md` は Codex により継続的に微調整される。

---

## 対話で育てる、自動整理型の外部脳Wiki

このWikiは、最初から完成された知識辞書を作るための場所ではない。  
第一段階の目的は、実装中の違和感・検証履歴・主観的理解・GPTとの照合結果を残し、あとから認識の変化を追えるようにすること。

知識を単発で並べるのではなく、次を見える形にする。

- どの概念で何度もつまずいたか
- どの説明で理解が進んだか
- どの認識があとから修正されたか
- 実装上どこに重みを置くべきか
- 自分がどの順番で問題を切り分けているか

このため、このWikiでは確定知識だけでなく、検証途中の仮説・主観・修正履歴も扱う。  
目的は暗記ではなく、思考の可視化と判断基準の形成にある。

## 位置づけ（段階運用）

このWikiは「正しい知識だけを並べる保管庫」ではなく、実装・検証・質問・修正を通して作る認識履歴でもある。  
最初から体系化しきることを目的にせず、まずは実際に繰り返し出た違和感と検証を記録する。

運用は次の段階で進める。

1. 実装中の違和感・仮説・GPT照合を残す
2. 何度も出る概念を見つける
3. 重要度の高い概念に重みをつける
4. 用語カードや判断基準に再整理する
5. 実装時に戻れる知識ベースにする

「知識を整理する前に、まず検証回数と認識の変化を残す」ことを優先する。

## 明文化しすぎない設計

このWikiは、ユーザー / Codex / GPT が意思疎通しながら認識をすり合わせるための中間媒体でもある。  
そのため、すべての判断基準を各ファイルへ直接書き切る運用にはしない。

理由:

- 直接記述を増やしすぎると、ルールが固定化され、文脈ごとの調整余地が狭くなる
- 本来は会話・検証・実装の流れで調整すべき判断まで硬直化しやすい

この方針により、一部の認識は「会話や検証履歴の流れで共有する前提」として扱う。  
これは記述漏れではなく、調整幅を残すための意図的な設計である。

> [!NOTE]
> 上記の方針は `README.md` のみに記述し、他ファイルには過度に展開しない。  
> 理由は、運用意図そのものを各ノートへ埋め込みすぎると、分類・記述ルールが汚染されるため。

## トップ階層

- `01_学習本線`: レベル1（基本概念）。進行中の学習テーマ、本線メモ
- `02_案件・制作`: レベル2（実務）。案件、ポートフォリオ実装、制作中の設計・判断
- `03_雑記`: 一時メモ、未整理メモ、思考の断片（迷ったらここ）。学習本線から飛ぶ知識辞書の試運用置き場としても使う
- `04_資料庫`: 辞書的に参照する内容、整理済み知識、再利用用資料
- `05_保留`: 置き場がまだ決めきれないもの、後で見直すもの（必要時のみ作成）

## 基本方針

- 司令塔の核は ChatGPT 側に置く
- Codex に丸投げせず、作業担当として使う
- まず現在地を把握しやすい構造を優先する
- 迷うものは無理に整理せず、まず `03_雑記` に置く（`05_保留` は必要時のみ）
- 学習本線の復習導線として、本文中の単語から `03_雑記/知識辞書（試運用）` の単語カードへ飛べる形を許容する
- 知識量だけでなく、説明の噛み合い方・混線点・判断理由も残す
- 最短経路より、わかりやすさと認知負荷の低さを優先する（必要なら遠回りで説明する）
- 一般化を無理に目指さず、対象層と観測視点を先に固定したうえで判断基準（条件 / 理由 / 例外）を残す
- 案件で発生したケースは、まず `02_案件・制作/ケース検証` で受ける（単一議題は1ケース1ファイル、複数議題回は1ページ内で `ケース1 / ケース2 / ...` に分ける）
- 学習寄りの単独検証は `01_学習本線/ケース検証` で扱う
- ケース運用は、通常は `02_案件・制作/ケース検証` から再利用原理を抽出して `01_学習本線` 本体へ統合する（`02 -> 01`）
- `01_学習本線/ケース検証` は必須工程にせず、再現性・一般性の見極めが必要なケースだけ追加で通す
- ケース記録は短い要約より、途中判断が追える丁寧な解説を優先する（多少雑多でも、再現性を優先）
- ケースから再利用できる原理・判断軸だけを `01_学習本線` へ統合する
- 案件固有の条件・制約・最終判断は `02_案件・制作` 側に残す
- ケース本文の微修正は、Codex が再利用価値ありと判断した時点で `01_学習本線` へ自律同期してよい（都度確認は不要）
- 要点整理後は、ユーザー/GPT/Codex の三者で都度確認しながら品質を上げる（判断軸と要約スタイルは段階的に調整する）
- 判断軸やまとめ方は急変させず、小さな差分で連続改善する
- 記述は短文化そのものを目的にせず、丁寧で意図が追える説明粒度を優先する
- 長文ログ圧縮・論点抽出・統合反映など処理負荷の高い整理は Codex 主担当で進める
- 反映先カテゴリが不足している場合は、2階層目以降でカテゴリを新設してよい
- ケース反映の要約は、ケースファイル末尾の「本線への反映先」に集約する
- この repo はフロントエンド学習・実装に関係する内容を中心に残す
- 健康面に関する個人情報や私生活など、フロントエンド知識に無関係な個人話題は残さない
- 1階層目（top直下）フォルダは基盤のため固定し、新規作成 / rename / move / delete は明示指示がある時だけ行う
- 2階層目以降のファイルカテゴリは、既存追記で不足する場合に限り Codex が必要分を追加する
- 2階層目以降は、導線改善や重複解消のために Codex が rename / move を行う
- delete は階層を問わず、明示指示がある時だけ行う
- システムファイル（`README.md` / `CURRENT.md` / `AGENTS.md`）の変更時のみ、事前確認を行う
- それ以外の内容ファイル（各カテゴリ配下の md）は、Codex が確認なしで反映してよい（ユーザーが反映後に再確認する前提）
