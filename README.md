## 対話で育てる、自動整理型の外部脳Wiki

このWikiは、最初から完成された知識辞書を作るための場所ではありません。  
第一段階の目的は、実装中の違和感・検証履歴・主観的理解・GPTとの照合結果を残し、あとから認識の変化を追えるようにすることです。

知識を単発で並べるのではなく、次の点を見える形にします。

- どの概念で何度もつまずいたか
- どの説明で理解が進んだか
- どの認識があとから修正されたか
- 実装上どこに重みを置くべきか
- 自分がどの順番で問題を切り分けているか

このため、このWikiでは確定知識だけでなく、検証途中の仮説・主観・修正履歴も扱います。
目的は暗記ではなく、思考の可視化と判断基準の形成にあります。

## 位置づけ（段階運用）

このWikiは「正しい知識だけを並べる保管庫」ではなく、実装・検証・質問・修正を通して作る認識履歴でもあります。  
最初から体系化しきることを目的にせず、まずは実際に繰り返し出た違和感と検証を記録します。

運用は次の段階で進めます。

1. 実装中の違和感・仮説・GPT照合を残します
2. 何度も出る概念を見つけます
3. 重要度の高い概念に重みをつけます
4. 用語カードや判断基準に再整理します
5. 実装時に戻れる知識ベースにします

「知識を整理する前に、まず検証回数と認識の変化を残す」ことを優先します。

## 明文化しすぎない設計

このWikiは、ユーザー / Codex / GPT が意思疎通しながら認識をすり合わせるための中間媒体でもあります。  
そのため、すべての判断基準を各ファイルへ直接書き切る運用にはしません。

理由:

- 直接記述を増やしすぎると、ルールが固定化され、文脈ごとの調整余地が狭くなります
- 本来は会話・検証・実装の流れで調整すべき判断まで硬直化しやすくなります

この方針により、一部の認識は「会話や検証履歴の流れで共有する前提」として扱います。  
これは記述漏れではなく、調整幅を残すための意図的な設計です。

> [!NOTE]
> 上記の方針は `README.md` のみに記述し、他ファイルには過度に展開しません。  
> 理由は、運用意図そのものを各ノートへ埋め込みすぎると、分類・記述ルールが汚染されるためです。

## 三者の性能を制限しないシステム運用

このWikiは、ユーザーがChatGPTやCodexの出力をすべて事前に理解し、完全に制御することを前提にしません。

ChatGPTやCodexには、ユーザー個人が把握しきれない知識の包括性・処理能力・整理能力があります。  
そのため、AIをユーザーの現在の理解範囲へ無理に縮小させるのではなく、それぞれが最大限の性能を発揮できる余地を残します。

ただし、AIへ判断を丸投げするわけではありません。  
出力・整理・反映の結果は、最終的にユーザーが確認し、必要に応じて修正します。  
そのため、ChatGPTやCodexにはユーザーへ過度に迎合させず、むしろ認識のズレ・不足・別視点を提示できる状態を維持します。

この方針を徹底するため、Codexについては最初からプロンプト設計や内部仕様の説明を前提にせず、あえて外部から見える対話上の挙動だけを観察して運用を構築しました。  
Codex自身がどこまで文脈を把握できるか、どの範囲まで自律的に判断できるか、どの指示で整理・反映・修正が安定するかを、リアルタイムの大量の対話と検証出力を通して確認しました。

つまり、このWikiはCodexの内部仕様を理解して制御するためのものではなく、ブラックボックスであるCodexの応答特性を観察し、実用上どこまで外部作業者として任せられるかを調整して作った運用システムです。  
ユーザーがすべてを事前に理解するのではなく、Codex自身の処理能力を活かしつつ、最終確認と判断境界だけをユーザー側に残す設計にしています。

この運用では、ユーザーの知識を単純に足し算で増やすだけではなく、ユーザー・ChatGPT・Codexの三者の役割を掛け合わせます。  
ユーザーは最終判断、ChatGPTは思考整理と方針確認、Codexは整理・統合・反映を担当します。  
この三者間の対話と検証を繰り返すことで、Wiki自体が比例的に成長するのではなく、判断精度・再利用性・整理能力が相互作用的に伸びる設計にしています。

## 用語の扱い

このWikiには、ChatGPTなどのLLMとの対話を通じて整理した性質上、
一般的な専門用語ではない表現や、個人的な造語が含まれる場合があります。

それらは、理解をつなぐための補助線であり、
必ずしも理屈やイメージそのものが誤っているという意味ではありません。

このWikiで造語や独自表現を使う目的は、
認知負荷を減らし、自分の中で概念を扱いやすくするためであります。

意思疎通を妨げることや、知識を誇示すること、
他者に対して優位に立つことを目的としたものではありません。

公式用語・一般的な専門用語・実務上の標準表現とは分けて確認し、
最終的には個人の責任で使用してください。

## トップ階層

- `01_学習本線`: レベル1（基本概念）として扱います。進行中の学習テーマと本線メモを置きます
- `02_案件・制作`: レベル2（実務）として扱います。案件、ポートフォリオ実装、制作中の設計・判断を置きます
- `03_雑記`: 一時メモ、未整理メモ、思考の断片を置きます（迷ったらここです）。学習本線から飛ぶ知識辞書の試運用置き場としても使います
- `04_資料庫`: 辞書的に参照する内容、整理済み知識、再利用用資料を置きます
- `05_保留`: 置き場がまだ決めきれないもの、後で見直すものを置きます（必要時のみ作成します）

## 推奨導線

- 実務先行で進める場合は、`02_案件・制作` → `01_学習本線` の順で進めます
- 概念から固める場合は、`01_学習本線` → `02_案件・制作` の順で進めます

## 基本方針

- まず現在地を把握しやすい構造を優先します
- 迷うものは無理に整理せず、まず `03_雑記` に置きます（`05_保留` は必要時のみ使います）
- 学習本線の復習導線として、本文中の単語から `03_雑記/知識辞書（試運用）` の単語カードへ飛べる形を許容します
- 知識量だけでなく、説明の噛み合い方・混線点・判断理由も残します
- 最短経路より、わかりやすさと認知負荷の低さを優先します（必要なら遠回りで説明します）
- 一般化を無理に目指さず、対象層と観測視点を先に固定したうえで判断基準（条件 / 理由 / 例外）を残します
- 案件で発生したケースは、まず `02_案件・制作/ケース検証` で受けます（単一議題は1ケース1ファイル、複数議題回は1ページ内で `ケース1 / ケース2 / ...` に分けます）
- ポートフォリオ全体の公開・記録範囲は `02_案件・制作/ポートフォリオサイト/00_公開・記録方針.md` で管理します
- 案件フォルダには、指示内容の詳細ではなく、自分が意識した実装・判断理由・工夫を残します
- 案件ログは、案件内の実践記録と、案件を超えて使えるHTML/CSS原理に分けて処理します
- 固有条件を外しても成立する原理は、重複を確認したうえで `01_学習本線` の既存ノートへ統合します
- 案件ファイル末尾の「学習本線へ戻した内容」から、一般化した要点と反映先を追跡できる形にします
- 規約の話で案件名が不明な場合は、普遍ルールへ一般化する前にどの案件か確認します
- 案件先の要望や既存の型は、その案件内の制約として扱い、HTML/CSS・BEMの共通ノートへ自動反映しません
- 学習寄りの単独検証は `01_学習本線/ケース検証` で扱います
- ケース運用は、通常は `02_案件・制作/ケース検証` から再利用原理を抽出して `01_学習本線` 本体へ統合します（`02 -> 01`）
- `01_学習本線/ケース検証` は必須工程にせず、再現性・一般性の見極めが必要なケースだけ追加で通します
- ケース記録は短い要約より、途中判断が追える丁寧な解説を優先します（多少雑多でも、再現性を優先します）
- ケースから再利用できる原理・判断軸だけを `01_学習本線` へ統合します
- 案件固有の条件・制約・最終判断は `02_案件・制作` 側に残します
- ケース本文の微修正は、Codex が再利用価値ありと判断した時点で `01_学習本線` へ自律同期して構いません（都度確認は不要です）
- 要点整理後は、ユーザー/GPT/Codex の三者で都度確認しながら品質を上げます（判断軸と要約スタイルは段階的に調整します）
- 判断軸やまとめ方は急変させず、小さな差分で連続改善します
- 記述は短文化そのものを目的にせず、丁寧で意図が追える説明粒度を優先します
- 長文ログ圧縮・論点抽出・統合反映など処理負荷の高い整理は Codex 主担当で進めます
- 反映先カテゴリが不足している場合は、2階層目以降でカテゴリを新設して構いません
- ケース反映の要約は、ケースファイル末尾の「本線への反映先」に集約します
- 運用変更と自律調整の恒久履歴は root の `CHANGELOG.md` に残します
- この repo はフロントエンド学習・実装に関係する内容を中心に残します
- フロントエンド知識に無関係なプライベートの個人話題は残しません
- 1階層目（top直下）フォルダは基盤のため固定し、新規作成 / rename / move / delete は明示指示がある時だけ行います
- 2階層目以降のファイルカテゴリは、既存追記で不足する場合に限り Codex が必要分を追加します
- 2階層目以降は、導線改善や重複解消のために Codex が rename / move を行います
- delete は階層を問わず、明示指示がある時だけ行います
- システムファイル（`README.md` / `CURRENT.md` / `AGENTS.md`）の変更時のみ、事前確認を行います
- それ以外の内容ファイル（各カテゴリ配下の md）は、Codex が確認なしで反映して構いません（ユーザーが反映後に再確認する前提です）
