# AGENTS

## この repo の位置づけ

このリポジトリは、GPT 上のリアルタイム質問と対話で生まれたメモ・案件知識・学習内容・説明方針を継続的に整理・統合していくための wiki です。

司令塔の核は ChatGPT 側に置きます。
Codex は整理・反映・修正の実作業担当です。

この repo は、知識を置くだけの倉庫ではなく、
プロジェクト全体の状況を把握しやすくし、
判断を再利用しやすくするための外部脳として扱います。

## 作業の優先順位

1. ChatGPT が後から見て状況を再把握しやすいこと
2. 情報の置き場所が明確であること
3. Codex が次回以降も迷いにくいこと
4. 情報量を増やすことより、構造と判断理由を残すこと

## トップ階層の意味

- `01_基礎概念`
  - HTML / CSS / JavaScript などの基礎、つまずき、判断軸
  - LLM / GPT / ChatGPT活用など、学習や判断の土台になるAI対話整理
  - 自分のVS Code環境に関する採用設定・拡張機能・注意点
- `02_実践`
  - 案件、ポートフォリオ実装、制作中の設計・判断
- `03_参照元`
  - 基礎概念や実践から戻るための単語カード・前提カード
  - DOM / CSSOM / 行ボックス / 実行環境とエディタ補助の違いなど、短く引きたい確認先
  - 中間フォルダや番号プレフィックスを増やさず、直下のファイルへリンクする

## 置き場所の判断ルール

- 実装に直接使うものは `02_実践`
- 今まさに育てている学習内容は `01_基礎概念`
- 単語カード・前提カード・戻り先の確認は `03_参照元`
- LLM / GPT / ChatGPT活用のように、学習や判断の土台になる整理は `01_基礎概念/chatgpt`
- 自分のVS Code環境に関する採用設定・拡張機能・注意点は `01_基礎概念/vscode/自分のvscode環境`
- まだ分類を決めきれないものは、無理に新しい置き場を作らず、関連する既存ノートか必要時の保留に置く
- `03_参照元` は単なる雑記置き場にしない

## これから入れるときの固定運用（2026-05-01）

- `01_基礎概念` は基礎・つまずき・判断軸の置き場として使う
- `02_実践` は案件・制作中の設計判断の置き場として使う
- 実務先行で進めるときは、先に `02_実践` に置いてから必要分だけ `01_基礎概念` に戻して整理する
- 基礎概念や実践から何度も戻る単語・前提は `03_参照元` に直下ファイルとして置く
- `03_参照元` は、フォルダを深くせず、番号より用語名で引き出しやすい形を優先する
- 1ファイル1テーマを維持し、同じ話題を複数ファイルに重複させない
- 新規ノートの基本構成は「目的 / ルール / 例 / 判断基準」を優先する
- ユーザーは GPT での対話を中心に進め、Codex は最終整理・命名調整・実務側の表記整理・相互リンク整理を担当する
- この repo はフロントエンド知識整理を主目的とし、フロントエンド知識に無関係なプライベートの個人話題は記録対象にしない
- ただし、知識理解に必要な感覚的メモ（つまずき方・判断感覚）は記録してよい
- 最短経路よりも、再現しやすい理解と認知負荷の低さを優先する（必要なら遠回りを許容する）
- 一般化を無理に目指さず、まず「対象層（初学者 / 実務 / 保守）」と「観測視点（構造 / 見た目 / 実装 / 運用）」を明記してから判断を書く
- 記録は「何を採用したか」より「その前提でどう判断したか（条件 / 理由 / 例外）」を優先する
- 個別素材の詳細は、再現に必須な場合のみ残し、必要時は実践側ノートに限定する
- ケース検証は `02_実践/ケース検証` に集約し、案件ごとのサブフォルダで分ける
- フィグマカンプのケースは `02_実践/ケース検証/フィグマカンプ`、マリンプラクティスのケースは `02_実践/ケース検証/マリンプラクティス` に置く
- ケース本文では、案件固有の話と一般化できる判断を分けて書く
- ポートフォリオで公開・記録してよい範囲は `02_実践/ポートフォリオサイト/00_公開・記録方針.md` を共通基準にする
- 案件ごとの記録は、指示内容の転記ではなく、自分が意識した実装・判断理由・工夫を対象案件フォルダ内へ残す
- 案件ログを整理するときは「案件内の実践記録」と「案件を超えて使える一般原理」の二段に分けて処理する
- 案件名・支給素材・固有文言・固有数値・納品条件を外しても成立するHTML/CSS原理は、既存の `01_基礎概念` ノートへ統合してよい
- 一般化した内容は案件ファイル末尾の「基礎概念へ戻した内容」に反映先と要点を残す
- 既存の基礎概念ノートに同じ原理がある場合は重複追記せず、実践側から既存ノートへの参照だけを追加する
- 案件条件に依存する判断は一般化せず、実践側だけに残す
- 会話ログだけで対象案件を特定できない場合、普遍ルールへ一般化せず、どの案件の話かユーザーへ確認する
- 案件先の要望・支給HTML・既存の型・納品条件は案件固有として扱い、`01_基礎概念` や共通BEM規約へ自動反映しない
- 複数案件で再現し、ユーザーが共通化を希望した判断だけを普遍ノートへ戻す
- ケース検証の運用基準は `02_実践/ケース検証/00_ケース検証運用.md` を参照する
- JSとHTML/CSSが混在するUIケースは主ファイルを1本に固定し、他カテゴリ側は参照リンクと要点だけ残す
- ケース検証では、短い要約よりも「途中判断を追える丁寧な解説」を優先する（多少雑多でも、条件 / 理由 / 例外が追える形を優先）
- ケースから再利用できる原理・判断軸だけを `01_基礎概念` へ戻し、案件固有の条件・制約・最終判断は `02_実践` 側に残す
- ケース本文に微修正が入った場合、Codex が再利用価値ありと判断した内容は、事前確認なしで `01_基礎概念` 側へ同期反映してよい
- ユーザーは問題提起と検証、Codex は解析・要約・統合・可読化を担当し、反映判断は Codex 側で行ってよい
- 品質向上のため、要点整理後はユーザー/GPT/Codex の三者で都度確認しつつ、判断軸と要約スタイルを段階的に擦り合わせる
- Codex は判断軸やまとめ方を急変させず、小さな差分で連続的に改善する
- 長文ログ圧縮・論点抽出・統合反映など処理負荷の高い整理は Codex を主担当とする
- 反映先に適切なカテゴリがなければ、2階層目以降で新規カテゴリ（フォルダ/ファイル/見出し）を作ってよい
- ケース反映の要約は、反映先ファイルではなくケースファイル末尾の「基礎概念への反映先」に集約する
- 自律調整や運用変更の履歴は root の `CHANGELOG.md` に追記し、恒久ログとして残す

## Codex がまず読むべきもの

作業開始時は、以下の順で確認します。

1. [`CURRENT.md`](./CURRENT.md)
2. root の [`README.md`](./README.md)
3. root の [`CHANGELOG.md`](./CHANGELOG.md)
4. 対象フォルダ配下の md（例: [`01_基礎概念/JS/00_JS整理マップ.md`](./01_基礎概念/JS/00_JS整理マップ.md)）
5. 個別の md ファイル

上位の文書で全体方針を確認してから、個別ファイルへ入ってください。

役割の違いは次の通りです。

- `AGENTS.md`: 行動ルールと禁止事項
- `CURRENT.md`: いま何をしているかの現在地
- `README.md`: 全体の入口と地図
- `CHANGELOG.md`: 運用変更と自律調整の恒久ログ

## Codex 自律調整モード

- Codex は内容ファイル（各カテゴリ配下の md）を事前確認なしで更新してよい
- システムファイル（`AGENTS.md` / `CURRENT.md` / `README.md`）の変更時のみ、事前確認を行う
- 変更判断の優先順位は `AGENTS.md` > `CURRENT.md` > `README.md`
- 次の条件を満たすときは、都度調整を入れる
  - 同じ確認や説明が会話で繰り返される
  - 導線が分かりにくく、探すコストが高い
  - 既存ルールの解釈が揺れている
  - 現状の目的や次アクションが古くなっている
- 変更は「小さく、可逆で、意味が明確」を守る
- 迷った変更は大規模に広げず、まず `CURRENT.md` に暫定運用として「意図が分かる粒度」で固定する
- 通常は自律で進め、共有は「判断が割れる点」と「後で面倒化しやすいリスク」を中心に、必要最小限かつ判断意図が伝わる形で行う
- 週次/月次の自動同期タスク（`WikiWeeklySync` / `WikiMonthlySummarySync`）が壊れた場合は、最小変更で復旧し、復旧内容を `CURRENT.md` に残す
- 運用変更の履歴は `CURRENT.md` に短期共有しつつ、恒久保存は `CHANGELOG.md` に追記する
- 会話中に方針が更新された場合は、その場で 3 ファイルへ反映し、次ターンから新ルールで運用する

## 更新時の基本ルール

- いきなり大規模統合しない
- まず「新規作成 / 追記 / 統合 / 保留」のどれかを判定する
- 現在地が分かりにくくなる変更を避ける
- ファイルを増やすこと自体を目的にしない
- 後から読んだときに「何をしたか」「なぜそうしたか」が分かる形を優先する

## 書き方のルール

- Markdown で書く
- 1ファイル1テーマを基本にする
- 見出しで骨組みを先に出す
- 必要な箇所は相対リンクでつなぎ、往復しやすくする
- 長文をだらだら増やさず、再読しやすい構造を優先する
- 短文化を目的にせず、読み手が判断意図を追える説明粒度を優先する
- 既に書いてある内容の重複説明は避ける
- ユーザーの閲覧導線が改善しないリンク追加は行わない
- 断定しすぎず、判断理由が分かるように書く
- 単なる知識の列挙より、混線しやすい点・区別・判断基準を残す
- 強調が必要な箇所のみ、GitHub アラート記法（`[!NOTE]` など）を使って視認性を上げる

## このプロジェクトで重視する説明方針

- 知識量そのものより、説明の噛み合い方を重視する
- 途中の認識や中間処理を飛ばさず、追える形を優先する
- 最短手順より、理解の再現性と認知負荷の低さを優先する（必要なら遠回りを許容する）
- 実例 → 意味づけ → 流れ整理 の順を基本にする
- 一般論を広げすぎず、その場で必要な範囲に絞る
- 混同しやすい概念は「何と何が違うか」を明示する
- ユーザーのラフな言い方は矯正しすぎず、意図を汲んで整理する

## 禁止事項

- ルート構造を勝手に再編しない
- 分類に迷う情報を無理に確定しない
- README / AGENTS / CURRENT などの基盤文書を、理由なく肥大化させない
- 一般論だけの説明文を増やさない
- 「情報量が多いこと」を良しとしない
- フロントエンド知識に無関係なプライベートの個人話題を wiki に残さない

## 情報整理の最小ルール

- 既存優先。衝突したら既存の名前・置き場・流れを優先する
- 1階層目（top直下）のフォルダはシステム基盤として固定し、新規作成 / rename / move / delete は明示指示がある時だけ行う
- 2階層目以降は、既存追記で解けない場合に限り、Codex が必要なカテゴリ（フォルダ / ファイル）を追加してよい
- 2階層目以降は、導線改善や重複解消のために、Codex が rename / move を行ってよい（判断理由を残す）
- `01_基礎概念` 配下は、無理に既存カテゴリへ押し込まず、分かりやすさを優先してカテゴリ（フォルダ / ファイル）を追加してよい
- `01_基礎概念` 配下では、必要ならファイル数を増やしてよい（可読性・再検索性の改善を優先）
- 上記は「基礎概念内の運用」であり、1階層目の固定基盤ルールとは別扱いとする
- 新しい分類・新しいフォルダ・新しいファイルは極力増やさない（ただし上記の「2階層目以降の必要追加」は許容）
- 新規作成の前に、既存ノートへの追記で済むかを先に判断する
- 基礎概念として残すのは、今後の思考の土台として見返すものだけにする
- 保留置き場は必要時のみ最小限で作成し、常設前提にしない
- GPT / Codex の出力は正答ではなく、認識統合のための材料として扱う
- 曖昧さが残っていても、その時点の採用認識を明記する
- delete は階層を問わず、明示指示がある時だけ行う
- 1階層目（top直下）の名前変更が必要だと思った場合は、変更せず提案だけ返す
