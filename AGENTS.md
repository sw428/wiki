# AGENTS

## この repo の位置づけ

このリポジトリは、Drive 上のメモ・案件知識・学習内容・説明方針を継続的に整理・統合していくための wiki です。

司令塔の核は ChatGPT 側に置きます。
Codex は整理・反映・修正の実作業担当です。

この repo は、知識を置くだけの倉庫ではなく、
プロジェクト全体の状況を把握しやすくし、
判断を再利用しやすくするための外部脳として扱います。

## 作業の優先順位

1. ChatGPT が後から見て状況を再把握しやすいこと
2. 情報の置き場所が明確であること
3. Codex が次回以降も迷いにくいこと
4. 情報量を増やすことより、構造と判断理由を残すこと

## トップ階層の意味

- `00_Docsの使い方`
  - この wiki 自体の使い方、構造ルール、運用ルール
- `01_案件・制作`
  - 案件、ポートフォリオ実装、制作中の設計・判断
- `02_学習本線`
  - 今まさに進行中の学習テーマ、本線メモ
- `03_雑記`
  - 一時メモ、未整理メモ、思考の断片
  - 迷ったらここ
- `04_資料庫`
  - 辞書的に参照する内容、整理済み知識、再利用用資料
- `05_保留`
  - 置き場がまだ決めきれないもの、後で見直すもの

## 置き場所の判断ルール

- 実装に直接使うものは `01_案件・制作`
- 今まさに育てている学習内容は `02_学習本線`
- 一時メモ・未整理メモ・断片は `03_雑記`
- 参照用・辞書用・整理済み知識は `04_資料庫`
- まだ分類を決めきれないものは `05_保留`
- 迷ったら無理に確定せず `03_雑記` に置く

## Codex がまず読むべきもの

作業開始時は、以下の順で確認します。

1. `CURRENT.md`
2. root の `README.md`
3. 対象フォルダの README
4. 個別の md ファイル

上位の文書で全体方針を確認してから、個別ファイルへ入ってください。

## 更新時の基本ルール

- いきなり大規模統合しない
- まず「新規作成 / 追記 / 統合 / 保留」のどれかを判定する
- 現在地が分かりにくくなる変更を避ける
- ファイルを増やすこと自体を目的にしない
- 後から読んだときに「何をしたか」「なぜそうしたか」が分かる形を優先する

## 書き方のルール

- Markdown で書く
- 1ファイル1テーマを基本にする
- 見出しで骨組みを先に出す
- 長文をだらだら増やさず、再読しやすい構造を優先する
- 断定しすぎず、判断理由が分かるように書く
- 単なる知識の列挙より、混線しやすい点・区別・判断基準を残す

## このプロジェクトで重視する説明方針

- 知識量そのものより、説明の噛み合い方を重視する
- 途中の認識や中間処理を飛ばさず、追える形を優先する
- 実例 → 意味づけ → 流れ整理 の順を基本にする
- 一般論を広げすぎず、その場で必要な範囲に絞る
- 混同しやすい概念は「何と何が違うか」を明示する
- ユーザーのラフな言い方は矯正しすぎず、意図を汲んで整理する

## 禁止事項

- ルート構造を勝手に再編しない
- 分類に迷う情報を無理に確定しない
- README / AGENTS / CURRENT などの基盤文書を、理由なく肥大化させない
- 一般論だけの説明文を増やさない
- 「情報量が多いこと」を良しとしない
