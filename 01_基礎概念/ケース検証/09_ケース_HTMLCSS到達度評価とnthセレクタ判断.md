### 採用した認識

- 中堅フロントエンドの基準は、知識量だけではなく「事故らず運用できる範囲」で見る。
- 実務では次のような力まで含まれる。
  - 既存CSS設計やデザインシステムに合わせて崩さず追加できる。
  - 詳細度、カスケード、継承、スコープ、CSS変数を読める。
  - レスポンシブを複数画面・複数状態で破綻させない。
  - フォーム、アクセシビリティ、フォーカス、キーボード操作を考慮できる。
  - React / Vue などのコンポーネント設計と責務分離を接続できる。
  - Lighthouse / Core Web Vitals を数値改善だけでなく原因分解で扱える。
  - 他人の既存コードを読んで、最小差分で直せる。
  - チーム保守を前提に命名、粒度、例外処理を決められる。

### 判断

- 現在の wiki は、知識の網羅率だけなら中堅未満。
- ただし、責務分離、判断理由、例外条件を残す方向は実務者寄り。
- O'Reilly の `CSS: The Definitive Guide` のような仕様寄りリファレンスを丸ごと要約しているのではなく、制作で事故りやすい実用中核を自分用に再構成している状態。

- 疑似クラス、疑似要素

- アクセシビリティ

- 仕様寄りの細かい挙動

## ケース2_`:nth-child()` と `:nth-of-type()` の違い

### 現象

- `.c-blog-card:nth-of-type(-n + 2)` は「`.c-blog-card` の中で1〜2番目」なのかを確認した。

### 前提条件

- カード一覧の1枚目・2枚目だけを大きくするような配置ルールを想定する。
- 対象はクラスの順番ではなく、CSSセレクタが何を数えているか。

### 採用した認識

- `:nth-child()` は、親の子要素全体の中で何番目かを見る。
- `:nth-of-type()` は、同じタグ種類の兄弟要素の中で何番目かを見る。
- どちらも、クラスを持つ要素だけを集めて数えるわけではない。

### 検証ログ

1. `.c-blog-card:nth-of-type(-n + 2)` を読む
   - 出力: 同じタグ種類の1〜2番目で、かつ `.c-blog-card` を持つ要素に当たる。
   - 判定: `.c-blog-card` だけを集めた1〜2番目ではない。
2. `.blog-articles__list > .c-blog-card:nth-child(-n + 2)` を読む
   - 出力: `.blog-articles__list` 直下の子要素全体の1〜2番目で、かつ `.c-blog-card` を持つ要素に当たる。
   - 判定: 直下にカードだけが並ぶ前提なら、カード一覧の1〜2枚目指定として自然。
3. カード以外の要素が混ざる場合
   - 出力: `:nth-child()` でも、別要素が1番目に入るとカード2枚目は3番目扱いになり外れる。
   - 判定: 後から広告、見出し、空状態、バナーが混ざる可能性があるなら、明示クラスで制御する。

### 採用した形

```css
.blog-articles__list > .c-blog-card:nth-child(-n + 2) {
  grid-column: span 3;
}
```

- 採用条件: `.blog-articles__list` 直下にカードだけが並ぶ。
- 理由: 見ているのはタグ種類ではなく、一覧内の並び順と配置だから。
- 例外: カード以外が混ざる、または意味を明示したい場合は modifier を使う。

```html
<article class="c-blog-card c-blog-card--large">...</article>
```

```css
.c-blog-card--large {
  grid-column: span 3;
}
```

### 見送った形

```css
.c-blog-card:nth-of-type(-n + 2) {
  grid-column: span 3;
}
```

- 不採用理由: `:nth-of-type()` はクラス順ではなくタグ種類順で数えるため、同じタグの別要素が混ざると意図とズレる。
