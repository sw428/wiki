# 06_スコープとクロージャ

## このファイルの役割

このファイルは、JavaScriptで「どこからどこまで見えるか」と「なぜ値が保持されるのか」を整理する場所。
特に、クロージャがなぜ起きるのかを、参照のつながりとして理解するための土台にする。

ここでは主に次を扱う。

- スコープ
- 外側と内側の関係
- クロージャ
- 参照の鎖
- 保持が解ける条件

## この範囲の芯

クロージャは、

`return` したから残る

ではなく、

参照され続けているから残る

と見ると整理しやすい。

つまり大事なのは次の3つ。

- 関数がどこで作られたか
- その関数が何を見に行けるか
- まだどこかから参照されているか

## スコープとは何か

スコープは、変数がどこから見えるかの範囲。

```js
function outer() {
  let x = 1;

  function inner() {
    console.log(x);
  }

  inner();
}
```

この例では、

- `x` は `outer` の中で作られる
- `inner` は `outer` の中で作られる
- だから `inner` は `x` を見に行ける

つまり、内側の関数は外側の変数を読める。

補足:
- スコープを主に見る対象は「変数宣言」
- 同じ `{}` でも、オブジェクトリテラルは「値の構造」であり、スコープ生成とは別

```js
console.log(name); // () は呼び出し用で、ここで新しいスコープは作らない

const obj = { name: "Taro" }; // {} はオブジェクト生成
```

## 外側と内側

JSでは、関数の内側から外側は見に行けるが、外側から内側のローカル変数は直接見えない。

```js
function outer() {
  let x = 1;

  function inner() {
    let y = 2;
    console.log(x, y);
  }

  inner();
}
```

このとき、

- `inner` は `x` を見られる
- `outer` の外から `y` は見られない

つまり、

内側は外側を見られるが、外側は内側を見られない

という向きがある。

## クロージャとは何か

クロージャは、
関数が、作られたときの外側の変数環境を一緒に持ち続ける仕組み。

```js
function outer() {
  let x = 1;

  return function () {
    return x++;
  };
}
```

このとき、外側の関数 `outer` は終わっていても、返された内側の関数が `x` を使い続けられる。

## `return inner` と `return inner()` は別

```js
function outer() {
  function inner() {
    return 1;
  }

  return inner;   // 関数そのものを返す（未実行）
  // return inner(); // 実行結果を返す
}
```

- `return inner`: 関数を返す
- `return inner()`: その場で実行して結果を返す

## よくある誤解

### 誤解

`return` したから保持される。

### 見方

参照が残っているから保持される。

つまり、`return` 自体が魔法ではない。
返された関数が、まだどこかから参照されているから、その関数が使う外側の値も残る。

## 参照の鎖

クロージャを考えるときは、参照の鎖で見るとわかりやすい。

```js
const fn = outer();
```

この状態では、

- `fn` が内側の関数を参照している
- 内側の関数が外側の変数 `x` を参照している

という鎖がある。

イメージで言うと、

- `fn`
- 内側の関数
- その関数が使う変数環境

がつながっている。

だから `x` はまだ消えない。

## 例

```js
function outer() {
  let x = 1;

  return function () {
    return x++;
  };
}

const f1 = outer();
const f2 = outer();

console.log(f1()); // 1
console.log(f1()); // 2
console.log(f2()); // 1
```

### 何が起きているか

- `outer()` を呼ぶたびに、新しい `x` が作られる
- `f1` は `f1` 用の `x` を持つ
- `f2` は `f2` 用の `x` を持つ

つまり、同じ形の関数でも、作られた回ごとに別の外側環境を持つ。

## なぜ値が残るか（コピーではない）

クロージャは「値コピー保存」ではなく、
返された関数が外側変数へ参照を持ち続けることで成立する。

```js
function createCounter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}

const counter = createCounter();
counter(); // 1
counter(); // 2
counter(); // 3
```

毎回作り直すと、毎回初期化される。

```js
createCounter()(); // 1
createCounter()(); // 1
```

## なぜ保持されるのか

関数がまだ参照されていると、その関数が必要とする外側の値も保持される。

逆に言うと、参照が切れれば保持も解ける。

```js
let fn = outer();

fn = null;
```

こうすると、

- `fn` から内側の関数への参照が切れる
- 内側の関数から外側環境への参照も辿れなくなる

この状態になると、不要になったものは回収対象になる。

## 保持が解ける条件の見方

ここで大事なのは、
もう使われない状態になったか
を見ること。

クロージャだから永遠に残るわけではない。
まだどこかから参照されている間だけ残る。

つまり、

- 参照がある -> 残る
- 参照が切れる -> 回収対象になる

という見方。

## クロージャがつながる話

クロージャを理解すると、次の話がつながる。

- イベントハンドラが変数を覚えている理由
- 状態を関数の中に閉じ込められる理由
- 古い値を持ち続ける問題
- メモリリークの入口

つまりクロージャは、単なる文法知識ではなく、関数が値を持ち続ける仕組みそのもの。

## イベントでも起きる見方

たとえばイベント処理でも、外側の値を内側の関数が使うことがある。

```js
function setupCounter() {
  let count = 0;

  const button = document.querySelector(".button");

  button.addEventListener("click", function () {
    count++;
    console.log(count);
  });
}
```

このとき、クリック時の関数は `count` を見に行っている。
だから `count` はその関数に必要な値として保持される。

## クロージャ変数と関数プロパティは別

### クロージャ変数

```js
function createCounter() {
  let count = 0;

  return function () {
    count++;
  };
}
```

- `count` は外側のローカル変数
- `counter.count` のように外から直接読めない

### 関数プロパティ

```js
function counter() {
  counter.count++;
}

counter.count = 0;
```

- `counter.count` で外から読める
- これは関数オブジェクトのプロパティ

同じ「count」でも、保存場所の種類が違う。

## この範囲の見方

クロージャは、
「内側の関数が外側の値を覚えている」
とだけ覚えるとぼやけやすい。

それより、

- 関数が参照されている
- 関数が外側の値を参照している
- その参照の鎖が残っている

と見たほうが、あとで混乱しにくい。

## この範囲のまとめ

- スコープは、変数がどこから見えるかの範囲
- 内側の関数は外側の変数を見られる
- クロージャは、関数が外側の変数環境を持ち続ける仕組み
- 保持される理由は `return` ではなく参照の継続
- 関数ごとに別の外側環境を持てる
- 参照が切れれば保持も解ける
- クロージャはイベントや状態管理にもつながる
