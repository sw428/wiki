# 07_実行環境とツール

## このファイルの役割

このファイルは、JavaScriptが「どこで動くのか」と、制作のために使う道具が「何の役割なのか」を整理する場所。
同じJavaScriptでも、動く場所や道具によってできることが変わる理由を見えるようにする。

ここでは主に次を扱う。

- ブラウザ
- Node.js
- VS Code
- Electron
- Prettier
- ESLint
- 拡張機能

## この範囲の芯

同じ JavaScript でも、

- どこで動くか
- 何のための道具か

で役割が変わる。

ここを整理する目的は、知識を増やすこと自体ではなく、
今自分が何をしていて、それが何のためなのかを見失わないこと。

## まず分けるもの

ここでは、次の3つを分けて考える。

1. 実行環境
JavaScriptが実際に動く場所。

2. 開発ツール
コードを書く、整える、確認するための道具。

3. 拡張機能や補助ツール
必要に応じて追加するもの。

## ブラウザ

ブラウザは、JavaScriptが動く実行環境のひとつ。

例:

- Chrome
- Edge
- Safari
- Firefox

ブラウザ環境では、主に次に関わる機能が使える。

- DOM
- CSSOM
- window
- 画面表示
- イベント処理

つまり、
HTML / CSS / 画面の世界を直接触れる実行環境
として見るとわかりやすい。

## Node.js

Node.js も、JavaScriptが動く実行環境のひとつ。

ブラウザと違って、主に次に強い。

- ファイル操作
- サーバー処理
- パッケージ管理
- ツール実行

つまり Node.js は、
画面よりも、実行や処理や道具を支える側の環境。

## ブラウザと Node.js の違い

### ブラウザが得意なもの

- DOM操作
- 画面表示
- ユーザー操作への反応

### Node.js が得意なもの

- ファイルを読む
- ツールを動かす
- サーバー処理をする

### 見方

同じ JavaScript でも、

- ブラウザで動かす JS
- Node.js で動かす JS

は、使える機能が同じではない。

だから
同じ言語 = 同じことができる
とは限らない。

## VS Code

VS Code は、JavaScriptの実行環境そのものというより、コードを書くための作業机。

ここでやるのは主に次。

- コードを書く
- ファイルを編集する
- 拡張機能を入れる
- 整形やLintを走らせる

つまり VS Code は、
JavaScriptそのものではなく、JavaScriptを書くための場所。

## Electron

Electron は、Node.js とブラウザ技術を組み合わせた土台。

VS Code は、この Electron の上で動いている。

つまり VS Code は、

- 中ではブラウザっぽい仕組みも使い
- Node.js 側の機能も使える

という、少し特殊なアプリ。

ここで大事なのは、
VS Code 自体が JavaScriptの基本仕様そのものではなく、
JavaScriptで作られた開発ツール側の世界だということ。

## なぜ全部ひとつじゃないのか

ここは学習上かなり大事。

同じ JavaScript でも、

- 実行環境
- ツール
- 拡張
- 設定

が分かれているのは、不便なだけではない。

分かれていることで、

- 役割が見える
- 今どこを触っているか分かる
- 学習の範囲を切り分けられる
- 必要なものだけ入れられる

という利点がある。

つまり、
全部入りで見えにくくするより、役割ごとに分けたほうが理解しやすい
という面がある。

## Prettier

Prettier は、コードの見た目を整える道具。

たとえば次を自動で整える。

- インデント
- スペース
- 改行
- クォート
- セミコロン

つまり Prettier は、コード整形ツール。

目的は、正しさそのものというより、
書式のばらつきを減らすこと。

## ESLint

ESLint は、コードのルール違反や危なさを見つける道具。

たとえば次をチェックする。

- 書き方のミス
- ルール違反
- バグの原因になりやすい書き方

つまり ESLint は、コード検査ツール。

Prettier と違って、見た目だけでなく
書き方の妥当性や危なさも見る。

## Prettier と ESLint の違い

### Prettier

- 見た目を整える
- 書式をそろえる

### ESLint

- ルール違反を見つける
- 危ない書き方を見つける

### 見方

- Prettier = 整形
- ESLint = 検査

この2つは似て見えるが、役割は違う。

## 拡張機能

拡張機能は、VS Code に後から道具を足す仕組み。

たとえば次。

- Prettier 拡張
- ESLint 拡張
- HTML / CSS 補助
- Git 補助

全部最初から入っていないのは、不便だからだけではなく、

- 人によって必要な道具が違う
- 軽く保ちたい
- 役割を分けたい

という理由もある。

## この話を先に整理する意味

制作で JavaScript を触るとき、最初は次のような小さな実務上の目的から始まることが多い。

- 演算子の前後にスペースを入れたい
- 保存時に整えたい
- エラーを減らしたい

でもそこから、

- Node.js
- 拡張機能
- ESLint
- Prettier
- 実行環境

へ話が広がる。

このとき大事なのは、
話が広がったこと自体ではなく、
何のためにそこを見ているのかを見失わないこと。

## 制作とのつながり

制作で最低限つながる流れはこう。

1. ポートフォリオ制作をしたい
2. JSを書く
3. 書式崩れや小さいミスで止まりたくない
4. Prettier や ESLint を使う
5. それを動かす土台として Node.js が出てくる
6. VS Code の拡張や設定が関係してくる

つまり、道具の話は本筋からそれた雑談ではなく、制作を進めるための周辺整理。

## Markdown運用（VS Code / Obsidian / GitHub）

学習ログを継続する場合は、次の役割分担が安定する。

- VS Code: 正確に編集する（検索/置換/差分確認）
- Obsidian: 読む・見返す（ノート導線）
- GitHub: バックアップ/履歴/共有

### 同期の実体

特別な同期機能ではなく、「同じローカルフォルダ」を両方で開く。

- VS Codeで保存
- 同じ `.md` をObsidianが読む

これで実質リアルタイム反映になる。

### Live Server との違い

- Markdownを読むだけなら VS Code標準プレビューで十分
- Live Server は基本HTML確認向け
- `.md` をサイト表示したい場合は HTML変換が必要

## 共有形式としてのMarkdown

Markdownは次に強い。

- 軽い
- 差分追跡しやすい
- ツール依存が低い
- 長期運用しやすい

ただし、非エンジニア向けに読みやすさ優先で渡す場合は
Google Docs / Notion / PDF の方が適する場面もある。

## Drive連携の前提（誤解防止）

「Driveを使っている」ことと「この場で自動読取できる」ことは別。

- 連携サービスがあっても、ここで扱えるのは渡された内容のみ
- 実運用は「ここで整理 → Driveに反映」の流れが安定する

## この範囲のまとめ

- JavaScriptは同じでも、動く場所で役割が変わる
- ブラウザは画面側の実行環境
- Node.js は処理やツール側の実行環境
- VS Code はコードを書くための作業机
- Electron は VS Code を支える土台
- Prettier は整形
- ESLint は検査
- 拡張機能は必要な道具を後から足す仕組み
- これらを分けて整理する目的は、今自分が何をしているかを見失わないため
