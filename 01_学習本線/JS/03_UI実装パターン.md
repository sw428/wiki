# 03_UI実装パターン

## このファイルの役割

このファイルは、JavaScriptでUIに動きをつけるときの基本パターンをまとめる場所。
制作でよく出る「開く・閉じる・切り替える・入力を確認する」を、共通の流れで整理する。

ここでは主に次を扱う。

- 開閉UI
- 切り替えUI
- フォーム確認
- 実例としてのハンバーガーメニュー
- 実例としてのアコーディオン

## この範囲の芯

UI実装の多くは、形が違っても中身はかなり似ている。

基本はこの流れ。

1. 必要な要素を取る
2. イベントを登録する
3. 今の状態を読む
4. class や属性や表示を変える
5. 見た目は CSS に任せる

つまり、

イベント -> 状態確認 -> 状態更新

が共通の芯。

## まず覚える型

### 基本形

```js
const button = document.querySelector(".button");
const target = document.querySelector(".target");

button.addEventListener("click", function () {
  target.classList.toggle("is-open");
});
```

ここでやっていることは次だけ。

- ボタンを取る
- 対象を取る
- クリックされたら
- `is-open` を付け外しする

かなり多くのUIは、この形から始まる。

## jQueryコードをバニラJSへ読み替える型

教材でjQueryが出る場合は、まず「何をしているか」を分解してから置換する。

### 例: ハンバーガーメニュー開閉

```js
$(function () {
  $(".js-menu").on("click", function () {
    $(".nav").toggleClass("is-open");
  });
});
```

上のコードは次の3段に分けて読める。

1. DOM準備後に実行する（`$(function () {})`）
2. `.js-menu` にクリックイベントを登録する（`.on("click", ...)`）
3. `.nav` の `is-open` を切り替える（`.toggleClass("is-open")`）

同じ処理をバニラJSで書くと次。

```js
document.addEventListener("DOMContentLoaded", () => {
  const menuButton = document.querySelector(".js-menu");
  const nav = document.querySelector(".nav");

  menuButton.addEventListener("click", () => {
    nav.classList.toggle("is-open");
  });
});
```

### 学習順（混線を防ぐ）

- 1回目: 教材通りに実装して、WordPress構造とUI挙動を再現する
- 2回目: 開閉やタブなど小さい処理からバニラJSへ置換する

最初から全面置換すると「CMS構造理解」と「JS変換」が同時進行になり、原因切り分けが難しくなる。

## UIでよく変えるもの

### class

```js
target.classList.add("is-open");
target.classList.remove("is-open");
target.classList.toggle("is-open");
```

見た目の変化を CSS 側で作っておき、JS は class の付け外しだけする。

### 属性

```js
button.setAttribute("aria-expanded", "true");
button.setAttribute("aria-expanded", "false");
```

見た目だけでなく、状態も属性で持たせると整理しやすい。

### 表示プロパティ

```js
target.hidden = true;
target.hidden = false;
```

`hidden` を使うと、開閉対象の表示を直接切り替えられる。

### 文字

```js
button.textContent = "閉じる";
```

必要なら、表示文字も状態に応じて変える。

## 開閉パターン

この型に入るもの:

- ハンバーガーメニュー
- アコーディオン
- モーダル

共通点は、
開いているか閉じているかを切り替えること。

## 切り替えパターン

この型に入るもの:

- タブ
- 表示モード切り替え
- アクティブ状態の切り替え

共通点は、
どれを active にするかを選ぶこと。

## 入力確認パターン

この型に入るもの:

- フォーム
- 検索欄
- バリデーション

共通点は、
入力値を読んで条件分岐すること。

## 実例: ハンバーガーメニュー

この例では、ハンバーガーボタンを押したときにメニューを開閉する。

### この例で取っている要素

- `.header`
- `.header__toggle`
- `#global-nav`
- `.header__nav-panel`

### 処理の分け方

この例では、処理を最初から2つの関数に分けている。

- `openMenu()`
- `closeMenu()`

こうしておくと、クリック時、Escapeキー、画面幅変更時など、どこからでも同じ処理を呼べる。

### 開くときにやっていること

`openMenu()` では次を行う。

- `header` に `is-open` クラスを付ける
- ボタンの `aria-expanded` を `"true"` にする
- ボタンの `aria-label` を「メニューを閉じる」に変える
- `nav` の `aria-hidden` を `"false"` にする
- `body` の `overflow` を `"hidden"` にして背景スクロールを止める

### 閉じるときにやっていること

`closeMenu()` では次を行う。

- `header` から `is-open` クラスを外す
- ボタンの `aria-expanded` を `"false"` にする
- ボタンの `aria-label` を「メニューを開く」に戻す
- `nav` の `aria-hidden` を `"true"` にする
- `body.style.overflow` を空文字に戻す

### クリック時の流れ

ボタンのクリック時には、まず

```js
header.classList.contains("is-open")
```

で今開いているかを調べる。

その結果が

- `true` なら `closeMenu()`
- `false` なら `openMenu()`

になる。

つまり、
状態を読んでから分岐する型になっている。

### ナビ領域クリック時の流れ

`nav` 全体にクリックイベントを付けて、

- パネル外をクリックしたか
- リンクをクリックしたか

を見て閉じている。

ここでは

- `panel.contains(e.target)`
- `e.target.closest("a")`

を使って判定している。

### Escapeキー対応

`document` 全体の `keydown` で、

- 押されたキーが Escape
- かつメニューが開いている

ときだけ `closeMenu()` を呼ぶ。

### 画面幅変更時の事故防止

`window` の `resize` で `matchMedia("(min-width: 769px)")` を使い、PC幅になったときは `closeMenu()` を呼ぶ。

これは、SPで開いたままPCに切り替わる事故を防ぐための処理。

### この例の芯

- 要素を取る
- 開く処理と閉じる処理を分ける
- クリック時に今の状態を読む
- class と aria 属性を一緒に変える
- 必要ならキーボード操作や画面幅変更にも対応する

## 実例: サービスカード開閉

この例では、各サービスカードの詳細本文をボタンで開閉する。

### この例で取っている要素

`.service-card__toggle` を `querySelectorAll()` で全部取る。

戻り値は `NodeList`。
配列ではないが、複数要素の集合として扱える。

### 複数要素への処理

`forEach()` を使って、各ボタンに同じクリック処理を登録している。

つまり、

1. ボタンを全部取る
2. 1個ずつ取り出す
3. 各ボタンにイベントを付ける

という流れ。

### ボタンを押したときの流れ

クリック時には、まず

```js
toggle.getAttribute("aria-controls")
```

で、対応する本文の id を文字列で取る。

次に

```js
document.getElementById(targetId)
```

で、その本文要素を取る。

### 今の状態の読み方

今開いているかは、

```js
toggle.getAttribute("aria-expanded")
```

を読んで、

- `"true"` なら開いている
- `"false"` なら閉じている

として判定している。

ここで重要なのは、`aria-expanded` の値は真偽値ではなく文字列だということ。

### 状態の反転

状態を反転するときは、

```js
!isOpen
```

で真偽値を反転し、

```js
String(!isOpen)
```

で `"true"` / `"false"` の文字列に変換してから `aria-expanded` に書き戻している。

### 本文の表示切り替え

本文側は

```js
body.hidden = isOpen
```

で切り替えている。

意味としては、

- 今開いていたなら `hidden = true` にして閉じる
- 今閉じていたなら `hidden = false` にして開く

という流れ。

### この例の芯

- ボタンと本文を `aria-controls` で対応づける
- `aria-expanded` で今の状態を読む
- 状態を反転して属性に書き戻す
- `hidden` で表示を切り替える

つまり、
対応先を属性でつなぎ、状態を属性と表示の両方に反映する型。

### 属性暗記より「作る順番」で固定する

開閉UIは毎回この順で作る。

1. 見せる本文を作る
2. 本文に `id` を付ける
3. 初期状態を `hidden` で閉じる
4. ボタンに `aria-controls` で相手を結ぶ
5. ボタンに `aria-expanded="false"` を置く
6. ボタン名（`aria-label` など）を付ける
7. JSで状態反転をつなぐ

### 4役で見る（最小記憶）

- 名前: `aria-label`
- 相手: `aria-controls`
- 状態: `aria-expanded`
- 実表示: `hidden`

この4問を毎回確認すると、属性の丸暗記より再現しやすい。

### 最小テンプレ（アイコンだけの開閉ボタン）

```html
<button
  class="service-card__toggle"
  type="button"
  aria-expanded="false"
  aria-controls="service-design-body"
  aria-label="Designの詳細を開閉"
>
  <span class="service-card__toggle-icon">▼</span>
</button>

<div id="service-design-body" hidden>
  ...
</div>
```

```js
const toggles = document.querySelectorAll(".service-card__toggle");

toggles.forEach((toggle) => {
  toggle.addEventListener("click", () => {
    const targetId = toggle.getAttribute("aria-controls");
    const body = document.getElementById(targetId);
    const isOpen = toggle.getAttribute("aria-expanded") === "true";

    toggle.setAttribute("aria-expanded", String(!isOpen));
    body.hidden = isOpen;
  });
});
```

## タブの基本形

タブは少しだけ段階が増える。

```js
const buttons = document.querySelectorAll(".tab-button");
const panels = document.querySelectorAll(".tab-panel");

buttons.forEach(function (button, index) {
  button.addEventListener("click", function () {
    buttons.forEach(function (btn) {
      btn.classList.remove("is-active");
    });

    panels.forEach(function (panel) {
      panel.classList.remove("is-active");
    });

    button.classList.add("is-active");
    panels[index].classList.add("is-active");
  });
});
```

### この型の見方

- いったん全部外す
- 押されたボタンだけ active にする
- 対応するパネルだけ active にする

つまり、
全解除 -> 必要なものだけ付与
という型。

## フォーム確認の基本形

```js
const form = document.querySelector(".form");
const input = document.querySelector(".form-input");
const message = document.querySelector(".form-message");

form.addEventListener("submit", function (event) {
  event.preventDefault();

  if (input.value === "") {
    message.textContent = "入力してください";
    return;
  }

  message.textContent = "送信できます";
});
```

### この型の見方

1. 送信時に処理する
2. まず既定の送信を止める
3. 入力値を読む
4. 条件分岐で表示を変える

## このファイル全体のまとめ

- UI実装の芯は `イベント -> 状態確認 -> 状態更新`
- 見た目は CSS に寄せ、JSは状態変更を担当する
- 開閉UIは「開いているか閉じているか」を切り替える
- 切り替えUIは「どれを active にするか」を選ぶ
- フォームは `value` を読んで `if` で分岐する
- 実例では class だけでなく aria 属性や hidden も一緒に変える
- 複雑に見えても、要素取得・イベント登録・状態更新に分けると追いやすい
