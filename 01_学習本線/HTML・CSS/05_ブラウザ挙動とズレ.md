# 05_ブラウザ挙動とズレ

## 目的

- 「なぜ画面がずれるか」を、レンダリング工程の因果で説明できるようにする。

## 基本フロー

`HTML解析 → DOM → CSSOM → Render Tree → Layout → Paint → Composite`

- この一連をレンダリングとして扱う。
- `Layout` は位置とサイズの計算工程。

補足:
- 画面に見える時点では結果がかなり出ている
- ただし内部では、計算順序と依存関係に沿って確定している

## 用語の関係

- Reflow: レイアウト再計算
- Layout Shift: 再計算の結果、表示位置が動く現象
- CLS: Layout Shift の累積指標
- Scroll Anchoring: Shift時に閲覧位置を保とうとするブラウザ補正

## 今回の主因パターン

`レスポンシブ切替 → Layout変更 → Reflow → Layout Shift → Scroll Anchoring補正 → 体感ズレ`

この鎖で起きることが多い。

## 画像が絡む時系列（重要）

画像は非同期ロードなので、次の順になりやすい。

1. 初期 `Layout` / `Paint`
2. JSでスクロール補正
3. 画像ロード完了
4. 画像サイズ確定で再 `Layout` / 再 `Paint`
5. 補正後の位置がまたずれる

つまり「JSが遅い」のではなく、
**JSの後に再レイアウト要因（画像確定）が来る**のが本質。

## スクロール時の見え方

- 単純スクロールは主に再描画（repaint）
- 高さが変わるイベントが混ざると再レイアウト（reflow）も発生
- 画像未予約だと、この再レイアウトが遅れて発火しやすい

## 防止策で何が変わるか

- `width` / `height` 属性: HTML段階で比率予約（最速）
- `aspect-ratio`: CSS適用段階で比率予約（実用上ほぼ同等）
- 予約後は、画像ロード時の高さ変化が減り、JS補正が壊れにくくなる

## 横幅変更が縦ズレを生む理由

- `max-width` や `padding` の変更は横方向の操作でも、テキスト折り返しを変える
- 折り返し行数が増減すると要素高さが変わる
- 結果として section 高さとスクロール位置がズレる

つまり「横幅の変更は直接ではなく、改行を介して縦方向に波及する」。

## CSS側で先に効く3点

1. `width` / `height` または `aspect-ratio` で画像高さを予約する
2. アンカーずれがあるなら `scroll-margin-top` を入れる
3. 画面外描画が重い場合は `content-visibility: auto` と `contain-intrinsic-size` を検討する

```css
main section[id] {
  scroll-margin-top: 80px;
}

main section {
  content-visibility: auto;
  contain-intrinsic-size: 800px;
}
```

補足:
- `content-visibility` は万能ではないため、要素特性やブラウザ差分を見ながら限定適用する。

## 対処順

1. 画像/動画のサイズ予約（`width`/`height`, `aspect-ratio`）
2. 高さが後から変わる要素を減らす（`fit-content` などを点検）
3. それでもズレる箇所に限定して `overflow-anchor: none;`
4. 最後に JS 補正を検討（CSS安定化後）

## 例

```css
.portfolio__list {
  overflow-anchor: none;
}
```

- 乱用せず、問題箇所だけに当てる。

## DevToolsでの見方（Computed / Overlay）

### Computedの値

- Computedは「最終採用された結果寄りの値」を見る場所
- 薄い灰色の `height: 439px` などは、CSSに直書きした値ではなく
  レイアウト計算の結果として表示されることがある

### Grid Overlay

- 紫の斜線は `gap` の可視化
- `gap` 数値はその場で直接表示されないことがある
- 空きトラック幅は `Show track sizes` で確認する

### Flexの `space-between`

- Gridのように「この空きが何px」と直接出しにくい
- 実務では `getBoundingClientRect()` で要素間を測る方が速い

## 初期認識メモ（確度）

- 確度: 高
  - 「Reflow が根にあり、Shift と Anchoring が連鎖する」という把握。
  - デバッグ順序を固定して学習効率を上げる視点。

## 判断基準

- まず CSS で再現性のある原因を潰す。
- `overflow-anchor` は最後の保険として使う。
