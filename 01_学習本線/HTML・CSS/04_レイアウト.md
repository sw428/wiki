# 04_レイアウト

## 目的

- Grid/Flex/Table/Float を「用途」と「副作用」で使い分ける。

## ルール

- `display` は継承しない。
- ただし、親が作るレイアウト文脈（grid/flex）は子の配置に影響する。
- Grid は親がトラック（列/行）を決める。子は親グリッドの列数を変更できない。

## Grid の要点

- `display: grid` の初期状態は実質1列運用（列未定義 + `grid-auto-flow: row`）。
- `gap` は「要素間」にだけ入る。先頭/末尾には入らない。
- 子側で `display: grid` を指定すれば、子の中に別グリッドを作れる（ネスト）。
- `li` が縦に並ぶのは `list-item` が勝ったからではなく、親Gridが1列のため。

## 通常フローとGridの見方

### 通常フロー

- 親の使える横幅が先に前提になる
- その中でテキスト/インラインが流れる
- 結果として高さが決まる

見え方としては「横幅先行、縦は結果」になりやすい。

### Grid

単純な空箱を順番に積むというより、

- 外側の制約
- トラックルール
- 中身の最小要求

を合わせて、どの列にどれだけ空間を割くかを解く。

見え方としては「先に領域計算を解く」感覚が強い。

### まとめ

親→子の物理時系列をそのまま見ているのではなく、
レイアウト計算の依存関係を理解しやすく時系列風に捉えている、と整理するとズレにくい。

### `fr` の扱い（要修正ポイントあり）

- `fr` は「残りスペースの比率分配単位」。
- `1fr 1fr` と `0.1fr 0.1fr` は比率が同じなので、列の取り分は同じ（1:1）。
- 「`0.1fr + 0.1fr` だから 20% しか使わない」は通常は誤り。

## `1fr` / `min-width: 0` / `minmax(0, 1fr)` の実務整理

- `1fr`: 単純に「残り全部」ではない。中身の最小要求で縮まないことがある
- `min-width: 0`: アイテム側の粘りを止める（子要素の縮み拒否を弱める）
- `minmax(0, 1fr)`: トラック側の粘りを止める（列の最小保証を弱める）

一言で:

- `min-width: 0` = 子が勝手に粘るのを止める
- `minmax(0, 1fr)` = 列が勝手に粘るのを止める

## Flex と Grid の実務使い分け

- 1方向の並び中心: Flex
- 行列を同時に設計: Grid
- 1列固定UI（カード縦積み）: Flexが最小で強い

### 1列固定（Flex最小形）

```css
.about__stats {
  display: flex;
  flex-direction: column;
  border: 1px solid var(--color-border);
  border-radius: 6px;
  overflow: hidden;
}

.about__stats > * + * {
  border-top: 1px solid var(--color-border);
}
```

### 複数列（Grid）

```css
.about__stats {
  display: grid;
  grid-template-columns: 1fr;
  border: 1px solid var(--color-border);
}

@media (min-width: 768px) {
  .about__stats {
    grid-template-columns: 1fr 1fr;
  }
}
```

## Table の扱い

- 表データはまず `<table>` を優先（意味構造が明確）。
- 列崩れが強い場合は `table-layout: fixed;` で安定化できる。
- ただしレスポンシブ柔軟性は下がるため、要件に応じて Grid/Flex へ置換検討。

## Float の位置づけ

- 旧来の回り込み用途。通常フローから外れる。
- 親高さ崩れや回り込み解除でハマりやすい。
- 新規レイアウトは原則 Flex/Grid 優先。

## Margin Collapse（マージン相殺）

block文脈では、上下マージンが合体することがある。

主な発生パターン:
1. 隣接兄弟（前要素の `margin-bottom` と後要素の `margin-top`）
2. 親子（子の `margin-top` が親の外側に見える）
3. 空要素（`margin-top` と `margin-bottom` が相殺）

止める代表手段:
- `padding`
- `border`
- `overflow`
- `display: flow-root`
- `display: flex`
- `display: grid`

## Formatting Context（レイアウト文脈）

| Context | 特徴 |
|---|---|
| Block | margin collapse が起きる |
| Flex | margin collapse は起きない |
| Grid | margin collapse は起きない |
| Inline | 行内配置ルールで処理 |

`display: flex/grid` は「新しいレイアウト文脈を作る指定」として見ると整理しやすい。

## Containing Block（位置・サイズ計算の基準）

- `width: 50%` や `position: absolute` の基準になる箱
- 「何を基準に計算しているか」がズレると、見た目が崩れやすい

## Gridでの `min-width: 0` と `minmax(0, 1fr)` の違い

どちらも「縮み拒否の強さ」を下げる指定だが、対象が違う。

| 指定 | 下げる対象 | 意味 |
|---|---|---|
| `min-width: 0` | グリッドアイテム | 子要素の「これ以上縮めない」主張を弱める |
| `minmax(0, 1fr)` | グリッドトラック | 列自体の最小保証を弱める |

実務では次の併用が多い。

```css
.layout {
  display: grid;
  grid-template-columns: 470px minmax(0, 1fr);
}

.layout__main {
  min-width: 0;
}
```

補足:
- `0` は「理論上そこまで縮んでよい」という計算上の下限
- 折り返し・省略・`overflow` の見え方は別で制御が必要

## Grid判断軸（実務での即決）

1. 同じものを並べるか  
   `repeat()` を使う。

```css
grid-template-columns: repeat(2, 1fr);
```

2. 左右で役割が違うか  
   列を直接指定する。

```css
grid-template-columns: 329px 1fr;
```

3. 見た目の順番だけ変えたいか  
   子に `grid-column` を当てる。

```css
.title {
  grid-column: 2;
}
```

## 親責任を先に固定する

レイアウト崩れの多くは「子でサイズを決める」ことから始まる。

- 親: トラック/幅/配置を決める
- 子: 中身を埋める

```css
/* NG: 子で直接サイズ調整 */
.item {
  width: 156px;
}

/* OK: 親で枠を作る */
.list {
  display: grid;
  grid-template-columns: repeat(2, 1fr);
}
```

## 幅が想定より広がる時の確認順

1. その要素の `display` は block か inline か
2. `width` / `max-width` をどこで止めているか
3. 親幅（グリッド列幅・コンテナ幅）が何で決まっているか
4. 子要素に `width: 100%` が入っていないか

補足:
- `aspect-ratio` は比率を決める指定であり、幅上限を止める指定ではない。
- 実寸上限を止める責務は `width` / `max-width` 側に置く。

## コンテナ責務の分離（本文幅とメディア幅）

- `l-inner` はページ全体幅、`l-narrow` は本文幅として責務を分ける。
- 見出し/本文は `l-narrow`、ヒーロー画像だけ広くしたい場合は `l-inner` 直下へ出す。
- 本文幅コンテナ内の要素を transform で無理に広げるより、コンテナ責務を分離した方が再利用しやすい。

## 認知アプローチ（知識に至る手順）

1. まず「同じ要素群か / 役割違いか」を判定する  
2. 親で構造を固定し、子での場当たり調整を止める  
3. 例外は全体ルールの後に最小限で追加する  
4. うまくいかないときは `min-width: 0` / `minmax(0, 1fr)` の順に確認する

## 初期認識メモ（確度）

- 確度: 高
  - 「Gridは親が全権、子は駒」という把握。
  - `display` 非継承 / 文脈影響ありの切り分け。
- 確度: 中
  - `gap` と `margin` の二重余白に注意する観点。
- 確度: 要修正
  - `fr` の合計が1未満だと余白が残る、は原則誤り。

## 判断基準

1. 1方向か2方向かで Flex/Grid を先に決める
2. 意味構造が表なら `<table>` を残す
3. Float は既存保守時のみ選択する


## 構造判断と余白責務の一般化

- SPで同一一覧として読めるなら、HTML骨格は維持し、PC差分はCSSで吸収する。
- 意味グループが分かれるときだけ `featured/list` のように構造分離する。
- 余白責務は次の3点で分ける。
  - 同種反復の間隔: `gap`
  - 別ブロック間の距離: `margin-bottom`
  - 線と中身の距離: `padding-top`
- 「見た目差」と「意味差」を分けて判断すると、構造の再編集コストを下げやすい。
