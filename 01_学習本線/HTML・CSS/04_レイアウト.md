# 04\_レイアウト

## 目的

- Grid/Flex/Table/Float を「用途」と「副作用」で使い分ける。

## このページの読み方（入り乱れ防止）

- まず骨格だけ読む:
  - `Grid の要点`
  - `Flex と Grid の実務使い分け`
  - `Table の扱い` / `Float の位置づけ`
- 次に詰まりやすい論点だけ読む:
  - `1fr / min-width: 0 / minmax(0, 1fr)`
  - `min-width: auto`
  - `flex-basis + padding`
- 後半は運用用:
  - `幅が想定より広がる時の確認順`
  - `認知アプローチ`

補足:

- 同じテーマ（`min-width` 系）が再登場するのは、概念説明と実務判断を分けるため。
- 重複ではなく「どの場面で使うか」を固定するための再掲として扱う。

## ルール

- `display` は継承しない。
- ただし、親が作るレイアウト文脈（grid/flex）は子の配置に影響する。
- Grid は親がトラック（列/行）を決める。子は親グリッドの列数を変更できない。

## `display` 非継承と文脈参加の切り分け

```css
.parent {
  display: grid;
}

.child {
  /* display: grid が継承されるわけではない */
}
```

- `.child` に `display: grid` が自動で入るわけではない（`display` は非継承）。
- ただし `.child` は、親 `.parent` が作ったグリッド文脈の中で「グリッドアイテム」として配置される。
- つまり「プロパティ値の継承」と「レイアウト文脈への参加」は別で考える。

## boxの分類（何を決めているか）

| 分類                | 何を決めるか                                          | 主な関連プロパティ                               |
| ------------------- | ----------------------------------------------------- | ------------------------------------------------ |
| 生成されるboxの種類 | block / inline / flex item など、どの文脈で扱われるか | `display`                                        |
| boxの寸法・位置     | 実際の幅・高さ・配置                                  | `width`, `height`, `line-height`, `grid`, `flex` |
| ボックスモデル      | `width` / `height` がどこまでを含むか                 | `box-sizing`, `padding`, `border`, `margin`      |

## Grid の要点

- `display: grid` の初期状態は実質1列運用（列未定義 + `grid-auto-flow: row`）。
- `gap` は「要素間」にだけ入る。先頭/末尾には入らない。
- 子側で `display: grid` を指定すれば、子の中に別グリッドを作れる（ネスト）。
- `li` が縦に並ぶのは `list-item` が勝ったからではなく、親Gridが1列のため。

### 反復要素の間隔は「親で束ねられるか」で選ぶ

同じ種類の要素が縦に続く時は、まずその集まりを親で束ねられるかを見る。

- 親で束ねられる一覧: `display: grid` / `display: flex; flex-direction: column;` + `gap`
- 既存構造のまま、2個目以降にだけ距離を付ける: 隣接兄弟の `margin-top`
- 次の別ブロックまでの距離を作る: セクションやブロック側の `margin-bottom`

```css
.article-list {
  display: grid;
  gap: 16px;
}

.article + .article {
  margin-top: 16px;
}
```

一律の `margin-bottom` は最後の要素にも効く。
そのため「同種反復の間隔」と「次の別ブロックまでの距離」を混ぜないようにする。

## `subgrid` の使いどころ

- 通常のネストGridは、親と子でトラック定義が独立する。
- `grid-template-columns: subgrid` / `grid-template-rows: subgrid` を使うと、親のトラック線を子に引き継げる。
- 向いている場面:
  - カード群で「見出し/本文/ボタン」の行位置をそろえたい
  - ネスト構造でも列線を統一したい
- 向いていない場面:
  - 単純な1段レイアウト（通常のGrid/Flexで十分）
- 導入前提:
  - 対象ブラウザ要件を確認し、非対応時の崩れ方を先に決める。

## 通常フローとGridの見方

### 通常フロー

- 親の使える横幅が先に前提になる
- その中でテキスト/インラインが流れる
- 結果として高さが決まる

見え方としては「横幅先行、縦は結果」になりやすい。

### Grid

単純な空箱を順番に積むというより、

- 外側の制約
- トラックルール
- 中身の最小要求

を合わせて、どの列にどれだけ空間を割くかを解く。

見え方としては「先に領域計算を解く」感覚が強い。

### まとめ

親→子の物理時系列をそのまま見ているのではなく、
レイアウト計算の依存関係を理解しやすく時系列風に捉えている、と整理するとズレにくい。

### `fr` の扱い（要修正ポイントあり）

- `fr` は「残りスペースの比率分配単位」。

## `1fr` / `min-width: 0` / `minmax(0, 1fr)` の実務整理

- `1fr`: 単純に「残り全部」ではない。中身の最小要求で縮まないことがある
- `min-width: 0`: アイテム側の粘りを止める（子要素の縮み拒否を弱める）
- `minmax(0, 1fr)`: トラック側の粘りを止める（列の最小保証を弱める）

一言で:

- `min-width: 0` = 子が勝手に粘るのを止める
- `minmax(0, 1fr)` = 列が勝手に粘るのを止める

## `min-width: auto` が文脈で違って見える理由

- 初期値としての `min-width` は `auto`。
- ただし実効値は文脈で変わる。
  - 通常フローのブロック文脈: 幅は親の利用可能幅基準で計算され、テキスト折り返しが先に効くため、`flex/grid` のような中身由来の縮み拒否としては見えにくい
  - flex/gridアイテム: `min-content` 側の制約として効きやすい
- 重要:
  - テキストノード自体に `min-width` プロパティはない。
  - 「縮まない」挙動は、親アイテムの自動最小サイズ規則として現れる。

## Flexで `flex-basis` と `padding` を同時に使う時

`flex: 1 1 391px` のような指定では、幅計算時にベースサイズへ左右paddingが加算される。
そのため横幅予算が厳しいと、内容領域が先に圧迫される。

このとき誤解しやすい点:

- `padding-left: clamp(...)` と `padding-right: 24px` は別プロパティで、優先順位競合ではない。
- 「右を増やしたら左が固定された」の主因はカスケード順ではなく、合計幅予算の圧迫。

実務での確認順:

1. `box-sizing` を確認する（`border-box` 前提か）
2. `flex-basis` を確認する（固定ベースが強すぎないか）
3. `min-width: 0` を必要なアイテムへ付与する
4. 長文には `overflow-wrap: anywhere;` を検討する

## Flex と Grid の実務使い分け

- 1方向の並び中心: Flex
- 行列を同時に設計: Grid
- 1列固定UI（カード縦積み）: Flexが最小で強い

### 入れ子のFlexはそれぞれ別の文脈

`align-items` や `justify-content` が配置するのは、そのFlexコンテナの直下にあるFlexアイテム。

```html
<div class="header-body">
  <h1 class="header-logo">...</h1>
  <div class="header-actions">
    <a href="#">電話</a>
    <a href="#">予約</a>
  </div>
</div>
```

```css
.header-body {
  display: flex;
  align-items: center;
  justify-content: space-between;
}

.header-actions {
  display: flex;
  align-items: center;
}
```

- `.header-body` が整列する対象は、直下のロゴと操作領域。
- `.header-actions` が整列する対象は、その直下のリンク。
- 外側の `align-items` は、孫要素であるリンク同士の配置までは決めない。
- 同じ指定を二箇所へ書いても、対象となるFlex文脈が違えば単純な重複ではない。

### flex item と中身の `display` は別で見る

親が `display: flex` の時、直下の子は外側の配置としては flex item になる。
その子自身に `display: grid` や `display: block` を指定しても、「親の中でflex itemとして扱われる」ことは変わらない。

```html
<div class="menu-row">
  <button class="hamburger-button">
    <span></span>
    <span></span>
    <span></span>
  </button>
</div>
```

```css
.menu-row {
  display: flex;
  align-items: center;
}

.hamburger-button {
  display: grid;
  gap: 6px;
}
```

- `.menu-row` から見る `.hamburger-button`: flex item
- `.hamburger-button` の中を見る: grid container

つまり、親の `align-items: center` はボタン自体を親の中で中央に置く指定。
子の `place-content: center` は、ボタン箱の中で線やアイコンを中央に置く指定。

ボタンに `width` / `height` がない場合、箱は中身の線幅・線高・`gap` で自然に決まる。
ただし実務では、見た目の線サイズと押せる領域を分けるため、ボタン側に `width` / `height` を明示する方が安定する。

### 固定サイド + 可変メインの2カラム

Flexで2カラムを作る時は、子要素へ幅を書くこと自体が悪いわけではない。
固定したい子と可変にしたい子を分ける。

```css
.page-layout {
  display: flex;
  gap: var(--layout-gap);
  align-items: flex-start;
}

.page-layout__main {
  flex: 1;
  min-width: 0;
}

.page-layout__side {
  width: var(--sidebar-width);
  flex-shrink: 0;
}
```

- 親: 横並び、gap、最大幅などの外枠を決める
- 固定側: 幅を持たせ、潰したくない場合は `flex-shrink: 0`
- 可変側: `flex: 1` で残り幅を受け、`min-width: 0` で縮み拒否を弱める

判断の芯:

- 固定したい子だけ幅を決める。
- 残りを受ける子はFlexに任せる。
- 両方を固定しすぎると、親幅が足りない時に横あふれしやすい。

### 縦Flexで最後の要素を下へ送る

カード本文の中で、最後のメタ情報だけ下へ押し下げたい場合は、本文領域を縦方向のFlexにする。

```css
.card__content {
  height: 100%;
  display: flex;
  flex-direction: column;
}

.card__meta {
  margin-top: auto;
  text-align: right;
}
```

- `margin-top: auto`: 余った縦スペースを上側marginで吸収し、要素を下へ送る
- `text-align: right`: 要素の中のテキストだけを右寄せする
- `align-self: flex-end`: flex itemである要素の箱そのものを右寄せする

`display: flex` を付けた要素自身が動くのではない。
その要素の直下の子要素が flex item として配置される。

そのため、テキストだけを右寄せしたいなら `text-align`、要素自身を押したいなら `margin` / `align-self`、複数の子要素を並べたいなら `flex` と分ける。

### 1列固定（Flex最小形）

```css
.about__stats {
  display: flex;
  flex-direction: column;
  border: 1px solid var(--color-border);
  border-radius: 6px;
  overflow: hidden;
}

.about__stats > * + * {
  border-top: 1px solid var(--color-border);
}
```

### 複数列（Grid）

```css
.about__stats {
  display: grid;
  grid-template-columns: 1fr;
  border: 1px solid var(--color-border);
}

@media (min-width: 768px) {
  .about__stats {
    grid-template-columns: 1fr 1fr;
  }
}
```

## Table の扱い

- 表データはまず `<table>` を優先（意味構造が明確）。
- 列崩れが強い場合は `table-layout: fixed;` で安定化できる。
- ただしレスポンシブ柔軟性は下がるため、要件に応じて Grid/Flex へ置換検討。

## Float の位置づけ

- 旧来の回り込み用途。通常フローから外れる。
- 親高さ崩れや回り込み解除でハマりやすい。
- 新規レイアウトは原則 Flex/Grid 優先。

## Margin Collapse（マージン相殺）

block文脈では、上下マージンが合体することがある。

主な発生パターン:

1. 隣接兄弟（前要素の `margin-bottom` と後要素の `margin-top`）
2. 親子（子の `margin-top` が親の外側に見える）
3. 空要素（`margin-top` と `margin-bottom` が相殺）

止める代表手段:

- `padding`
- `border`
- `overflow`
- `display: flow-root`
- `display: flex`
- `display: grid`

## Formatting Context（レイアウト文脈）

| Context | 特徴                       |
| ------- | -------------------------- |
| Block   | margin collapse が起きる   |
| Flex    | margin collapse は起きない |
| Grid    | margin collapse は起きない |
| Inline  | 行内配置ルールで処理       |

`display: flex/grid` は「新しいレイアウト文脈を作る指定」として見ると整理しやすい。

## Containing Block（位置・サイズ計算の基準）

- `width: 50%` や `position: absolute` の基準になる箱
- 「何を基準に計算しているか」がズレると、見た目が崩れやすい

## Gridでの `min-width: 0` と `minmax(0, 1fr)` の違い

どちらも「縮み拒否の強さ」を下げる指定だが、対象が違う。

| 指定             | 下げる対象       | 意味                                     |
| ---------------- | ---------------- | ---------------------------------------- |
| `min-width: 0`   | グリッドアイテム | 子要素の「これ以上縮めない」主張を弱める |
| `minmax(0, 1fr)` | グリッドトラック | 列自体の最小保証を弱める                 |

実務では次の併用が多い。

```css
.layout {
  display: grid;
  grid-template-columns: 470px minmax(0, 1fr);
}

.layout__main {
  min-width: 0;
}
```

補足:

- `0` は「理論上そこまで縮んでよい」という計算上の下限
- 折り返し・省略・`overflow` の見え方は別で制御が必要

## Grid判断軸（実務での即決）

1. 同じものを並べるか  
   `repeat()` を使う。

```css
grid-template-columns: repeat(2, 1fr);
```

2. 左右で役割が違うか  
   列を直接指定する。

```css
grid-template-columns: 329px 1fr;
```

3. 見た目の順番だけ変えたいか  
   子に `grid-column` を当てる。

```css
.title {
  grid-column: 2;
}
```

## 親責任を先に固定する

レイアウト崩れの多くは「子でサイズを決める」ことから始まる。

- 親: トラック/幅/配置を決める
- 子: 中身を埋める

```css
/* NG: 子で直接サイズ調整 */
.item {
  width: 156px;
}

/* OK: 親で枠を作る */
.list {
  display: grid;
  grid-template-columns: repeat(2, 1fr);
}
```

## 幅が想定より広がる時の確認順

1. その要素の `display` は block か inline か
2. `width` / `max-width` をどこで止めているか
3. 親幅（グリッド列幅・コンテナ幅）が何で決まっているか
4. 子要素に `width: 100%` が入っていないか

補足:

- `aspect-ratio` は比率を決める指定であり、幅上限を止める指定ではない。
- 実寸上限を止める責務は `width` / `max-width` 側に置く。

## inline要素の背景面積（参照）

- `display: inline` の背景面積と「背景が見えない時の確認順」は、[03\_画像と背景](./03_画像と背景.md) を参照。

## コンテナ責務の分離（本文幅とメディア幅）

- `l-inner` はページ全体幅、`l-narrow` は本文幅として責務を分ける。
- 見出し/本文は `l-narrow`、ヒーロー画像だけ広くしたい場合は `l-inner` 直下へ出す。
- 本文幅コンテナ内の要素を transform で無理に広げるより、コンテナ責務を分離した方が再利用しやすい。

### 全幅の外側と、幅制限する内側

背景色・罫線を画面幅まで伸ばし、中身だけ中央へ収めたい場合は、外側と内側を分ける。

```html
<div class="section-band">
  <div class="section-inner">
    ...
  </div>
</div>
```

```css
.section-band {
  border-bottom: 1px solid #ccc;
  background-color: #eef6ff;
}

.section-inner {
  width: min(calc(100% - 32px), 1080px);
  margin-inline: auto;
}
```

- 外側: 背景・線・全幅領域
- 内側: `max-width` 相当の幅制限・中央寄せ・左右の安全余白
- カンプの左右余白を大きな固定 `padding-inline` へ直訳する前に、「中身の最大幅」を表す数値ではないか確認する。
- 線だけを広くしたい場合は、線を外側へ持たせる。

## 認知アプローチ（知識に至る手順）

1. まず「同じ要素群か / 役割違いか」を判定する
2. 親で構造を固定し、子での場当たり調整を止める
3. 例外は全体ルールの後に最小限で追加する
4. うまくいかないときは `min-width: 0` / `minmax(0, 1fr)` の順に確認する

## 初期認識メモ（確度）

- 確度: 高
  - 「Gridは親が全権、子は駒」という把握。
  - `display` 非継承 / 文脈影響ありの切り分け。
- 確度: 中
  - `gap` と `margin` の二重余白に注意する観点。

## 判断基準

1. 1方向か2方向かで Flex/Grid を先に決める
2. 意味構造が表なら `<table>` を残す
3. Float は既存保守時のみ選択する

## 構造判断と余白責務の一般化

- SPで同一一覧として読めるなら、HTML骨格は維持し、PC差分はCSSで吸収する。
- 意味グループが分かれるときだけ `featured/list` のように構造分離する。
- 余白責務は次の3点で分ける。
  - 同種反復の間隔: `gap`
  - 別ブロック間の距離: `margin-bottom`
  - 線と中身の距離: `padding-top`
- 「見た目差」と「意味差」を分けて判断すると、構造の再編集コストを下げやすい。
