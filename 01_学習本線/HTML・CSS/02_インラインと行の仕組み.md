# 02\_インラインと行の仕組み

## 目的

- インライン要素の高さ・基準線・ズレ方を説明できるようにする。

## この話をする意図（言葉遊びにしないため）

- 目的は用語暗記ではなく、表示崩れを「どの段階で起きたか」に分解して修正速度を上げること。
- `display` / box / テキストノード / 継承 を分けると、次の切り分けができる。
  - CSSを書いたのに見えない: 指定ミスなのか、boxの面積不足なのか
  - 中央寄せがずれる: `text-align` の対象要素が違うのか、line box / baseline 側の問題か
  - 画像下に隙間が出る: 画像自体の問題か、行内文脈（baseline）由来か
- つまりこの章は「概念説明」ではなく、**実装時に再現性のある確認順を作るための土台**。

## ルール

- 行内レイアウトは「[行ボックス](../../03_雑記/知識辞書（試運用）/02_単語カード/行ボックス.md)」と「[ベースライン](../../03_雑記/知識辞書（試運用）/02_単語カード/ベースライン.md)」を基準に並ぶ。
- `line-height` は [行ボックス](../../03_雑記/知識辞書（試運用）/02_単語カード/行ボックス.md) の高さを決める中心パラメータ。
- `inline-block` は外側では行内要素として [行ボックス](../../03_雑記/知識辞書（試運用）/02_単語カード/行ボックス.md) に参加し、内側では `block` 的に `width` / `height` / `padding` を扱える。
- 文字自体は [テキストノード](../../03_雑記/知識辞書（試運用）/02_単語カード/テキストノード.md) で、指定反映は [CSS適用境界](../../03_雑記/知識辞書（試運用）/02_単語カード/CSS適用境界.md) を分けて確認する。

## 例

```css
p {
  font-size: 16px;
  line-height: 1.5; /* 行ボックスは 24px 相当 */
}

.badge {
  display: inline-block;
  padding: 2px 8px;
  background: #eef;
  vertical-align: middle;
}
```

- `vertical-align` は行内での基準合わせに効く。
- 画像や `inline-block` の下に隙間が見えるときは、[ベースライン](../../03_雑記/知識辞書（試運用）/02_単語カード/ベースライン.md) 整列が原因になりやすい。

## フォント中心と視覚中心のズレ

- フォントには `ascender` / `descender` / `baseline` のメトリクスが常にある。
- 文字が大文字だけでも、フォントメトリクス上の下側領域は残る。
- そのため「数学的中央」と「視覚的中央」は一致しないことがある。

実務では次の微調整が普通に発生する。

```css
.button__label {
  display: inline-flex;
  align-items: center;
  transform: translateY(1px);
}
```

## どの箱を基準に揃えているか

　　(今覚える必要はなし、軽く触れておくことで今後レンダリングフロー意識しやすくなる)

| 名称                 | 正体                         | 主に効く場所       |
| -------------------- | ---------------------------- | ------------------ |
| inline box           | 行内に流れる箱               | 行内での並び       |
| inline-block box     | 行内に置ける矩形箱           | 行内の矩形UI       |
| font box             | フォント内部の仮想箱         | 文字の描画         |
| glyph box            | 個々の字形の描画領域         | 文字の見え方       |
| line box             | 行全体の箱（`line-height`）  | 行内配置           |
| replaced element box | `img` など外部中身を持つ箱   | 画像・動画の配置   |
| border box           | `content + padding + border` | レイアウト単位     |
| flex item box        | flex配置対象の箱             | `align-items` など |

## box は `display` の言い換えではない

- `display` は CSS の設定値であり、書いた時点では「どう描画されるかのルール」を与えている段階。
- `box` はそのルールをもとに描画時に生成される結果で、ここで初めてレイアウト対象が具体化される。
- その後に `layout` で box のサイズ/位置が計算され、`paint` で背景・文字・画像が見える形になる。
- つまり `display` と `box` は同義語ではなく、**設定と結果**の関係で分けて見る。

流れを固定すると、次の順で追える。

1. DOM に要素があるか
2. その要素が `display` でどの性質の box になるか
3. `layout` で幅・高さ・位置が発生しているか
4. `paint` される領域があるか

補足:

- box は背景が付く矩形だけでなく、`line box` / `font box` / `glyph box` のような計算用の箱も含む。
- 「CSSが効いていない」に見える不具合も、実際は box の種類や面積の問題で説明できることが多い。

## `inline-block` と `inline-flex` の違い

- `inline-block`: 外側は行内文脈（[行ボックス](../../03_雑記/知識辞書（試運用）/02_単語カード/行ボックス.md) / [ベースライン](../../03_雑記/知識辞書（試運用）/02_単語カード/ベースライン.md)）に参加し、内側は `block` 的に扱う
- `inline-flex`: 外側は行内文脈に参加しつつ、内側の子要素配置は `align-items` など flex 規則で制御できるため、縦揃えを安定させやすい

使い分けの芯:

- `inline-block`: 文字の流れの中に置ける「小さい箱」
- `inline-flex`: 文字の流れの中に置ける「中身を整列できる小さい箱」

### 単純なボタンは `inline-block`

`a` / `span` など、初期状態が `inline` の要素に `padding` / `width` / `height` を効かせて、文字の流れに置けるボタン箱にしたい場合は `inline-block` で足りる。

```html
<a class="btn" href="#">View Recipe</a>
```

```css
.btn {
  display: inline-block;
  padding: 18px 60px;
  border: 1px solid #2b2a27;
  text-decoration: none;
}
```

- 文字だけ + `padding` の普通のボタンなら、必要なのは「外側の箱化」。
- 文字が裸のままという意味ではなく、文字の周囲に `padding` でボタン面積を作る。
- 中に整列したい複数要素がなければ、`inline-flex` まで使わなくてよい。

### アイコン付きボタンは `inline-flex`

ボタン内にアイコン、矢印、複数の `span` などがあり、中央揃えや `gap` が必要な場合は `inline-flex` が向いている。

```html
<a class="btn" href="#">
  <span class="btn__text">View Recipe</span>
  <span class="btn__icon">→</span>
</a>
```

```css
.btn {
  display: inline-flex;
  align-items: center;
  justify-content: center;
  gap: 8px;
  padding: 18px 60px;
  border: 1px solid #2b2a27;
  text-decoration: none;
}
```

- 外側は行内文脈に置ける。
- 内側は flex 文脈になり、子要素同士の縦揃えや間隔を制御しやすい。
- `gap` / `align-items` / `justify-content` が判断材料になる。

### CTAボタンでの実務使い分け

`inline-block` と `inline-flex` は、どちらも外側としては行内に置ける。
違いは、内側の中身をどう扱いたいか。

- 文章中の小さいリンク・ラベル:
  - `inline-block`
  - テキストの流れやベースライン感を残しやすい
- 独立したCTAボタン:
  - `inline-flex` でも `inline-block` でもよい
  - 文字だけなら `inline-block` で足りる
- 文字 + 矢印 / アイコン:
  - `inline-flex`
  - `align-items` / `justify-content` / `gap` で内部整列しやすい

```css
.button-link {
  display: inline-flex;
  align-items: center;
  justify-content: center;
  gap: 18px;
  padding: 11px 16px 12px 20px;
}
```

ボタン面積は、まず `padding` で作る。
カンプ上の最低サイズを守りたい時だけ `min-width` / `min-height` を足す。
完全固定が要件なら `width` / `height` を使うが、文字変更には弱くなる。

### `::after` とpaddingの関係

`::after` の `content` は、元の要素の中に追加される疑似的な子要素として考える。

```css
.button-link {
  display: inline-flex;
  align-items: center;
  gap: 18px;
  padding-inline: 20px 16px;
}

.button-link::after {
  content: ">";
}
```

この場合、`::after` はpaddingの外へ出るのではなく、paddingの内側に入る。

```txt
[左padding][文字][gap][::after][右padding]
```

単純な文字矢印なら `::after` で足りる。
SVG画像や細かい位置・サイズ調整が必要なら、実要素の `span + img` に分ける。

補足:

- フォントメトリクスの影響が「ゼロ」になるわけではない
- 変わるのは主に「揃えの基準」

## 画像リンクで起きる基準線由来の下隙間

- `inline-flex` は「内側がflex」でも「外側はinline」のまま。
- そのため、画像リンクのような要素では baseline/descender 由来の下隙間が残ることがある。
- 画像枠を安定させたいときは、次を基準にする。

```css
.media-link {
  display: block;
}

.media-link > img {
  display: block;
}
```

- `display: flex` で隙間が消えるケースもあるが、画像1枚リンクなら `block` の方が意図が明確で副作用が少ない。

## テキストノードとタグ要素の分離

- HTML の文字そのものは [テキストノード](../../03_雑記/知識辞書（試運用）/02_単語カード/テキストノード.md) として存在する。
- `p` / `span` / `button` などは、その文字を包むタグ要素。
- CSS の指定先は基本的に要素であり、テキストノードへ直接指定するわけではない。
- ただし要素へ指定した継承プロパティや整列ルールが、結果としてテキスト描画に使われる。

| 言い方                 | 指しているもの                       |
| ---------------------- | ------------------------------------ |
| テキストを直接包む要素 | `p` / `span` / `button` など         |
| テキストノードの親要素 | テキストノードを直接子として持つ要素 |
| 要素の親要素           | そのタグのさらに外側の要素           |

「親要素」という言葉は、文脈で指す対象が変わるため混線しやすい。
このノートでは次を固定する。

- テキスト配置（`text-align` など）を説明するときは、まず **テキストノードを直接包む要素** を起点にする。
- さらに外側の親要素は、幅制約やレイアウト文脈（`flex` / `grid`）を見る段階で分けて扱う。

```html
<p class="lead">Hello</p>
```

```css
.lead {
  text-align: center;
  color: red;
  font-size: 16px;
}
```

- CSS が当たる先は `.lead` 要素。
- `Hello` はテキストノード。
- `.lead` は `Hello` から見た親要素（テキストを直接包む要素）。
- `text-align: center` は、`.lead` の内容領域内でインライン内容を中央寄せするルールとして働く。

```html
<div class="card">
  <p class="card__text">Hello</p>
</div>
```

```css
.card__text {
  text-align: center;
}
```

- この場合、`text-align` が指定されているのは `.card` ではなく `.card__text`。
- `Hello` から見た親要素は `.card__text` なので、中央寄せも `.card__text` の内容領域基準で起きる。
- その外側の `.card` は、必要なら幅やレイアウト文脈を見る対象として別に確認する。

## `text-align` と中央寄せ責務の分離

`center` 系指定は「何を中央にするか」を分けて考える。

- `text-align: center;`  
  テキストノードを直接包む要素の内容領域内で、インライン内容（テキスト、inline要素）を中央寄せする。
- `margin-inline: auto;`  
  ブロック要素の箱そのものを親の中で中央に寄せる。
- `display: flex; justify-content: center;`  
  flex文脈で子要素を主軸中央に寄せる。

補足:

- `text-align` はテキストノードに直接指定するのではなく、テキストノードを含む要素の整列ルールとして効く。
- その結果として文字が中央に「見える」。

## 継承プロパティとテキスト描画

- テキスト描画は、要素に指定された継承プロパティが子孫へ渡ることで成立する。
- 分離して見ると、次の3段階で説明できる。

| 区分              | 何が起きているか                       | 代表例                                |
| ----------------- | -------------------------------------- | ------------------------------------- |
| CSS の直接対象    | 指定は要素に対して解釈される           | `.lead { color: red; }`               |
| 継承/計算の通り道 | DOM 親子関係を通って値が伝わる         | `font-size` / `line-height`           |
| 描画時の影響対象  | テキストノードや字形描画に値が使われる | `glyph box` / `font box` / `line box` |

- 代表例:
  - `font-size`
  - `font-family`
  - `font-weight`
  - `line-height`
  - `color`
  - `letter-spacing`
- フォントメトリクスはこの中の一部概念（基準線や字面寸法）として扱う。
- `text-align` は文字へ直接貼る指定ではなく、テキストを含む要素内のインライン内容をどこに配置するかを決める。

## 初期認識メモ（確度）

- 確度: 高
  - `line-height` が行の見え方に強く効く。
  - `inline-block` は行内文脈を持ちながら `width`/`height` が扱える。
- 確度: 中
  - 「中身の有無で基準の見え方が変わる」という観察（主因はベースライン整列と行ボックス）。
- 確度: 要修正
  - 「テキストがあれば左上基準、なければ baseline 基準」と断定するより、常に行内フォーマット文脈での整列として扱うほうが再現性が高い。

## 判断基準

- 行内ズレを見たら次を確認する。

1. `line-height`
2. `vertical-align`
3. 置換要素（`img` など）かどうか
4. `display: inline` / `inline-block` の違い
5. 対象が「文章の一部」か「独立したUI部品」か
6. 指定している要素が、テキストを直接包む要素か、その外側か
