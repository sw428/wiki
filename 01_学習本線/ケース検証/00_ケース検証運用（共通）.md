# 00_ケース検証運用（共通）

## 目的

- ここに貼られた内容を、まずケース検証ファイルとして形成して履歴化する。
- その後、再利用価値のある原理だけを各カテゴリ本線へ戻す。
- 「正解集」ではなく「条件 -> 出力 -> 採用理由」の検証ルートを残す。

## 基本フロー（固定）

1. まず `01_学習本線/ケース検証` に受け取り回を記録する（単一議題は1ケース1ファイル、複数議題回は1ファイル内で `ケース1 / ケース2 / ...` に分割する）  
2. 各ケースごとに現象/条件/出力/採用理由を独立させて確定する  
3. 各カテゴリ本線（HTML・CSS / JS / React / TypeScript）へ必要知識だけ反映する（日時ログ追記ではなく、カテゴリ見出しへ一般化して統合する）  
4. ケースファイル末尾の「本線への反映先」を更新し、反映内容を集約する

## ファイル命名

- `00`: 運用ルール専用（このファイル）
- `01` 以降: `NN_ケース_<短いケース名>.md`
- 例:
  - `01_ケース_Blogカード画像_レンダリング挙動.md`
  - `02_ケース_aria-hiddenとモーダル状態同期.md`

## 1ファイルに書く範囲

- 単一議題の回は、1ファイル1状況に限定する。
- 複数議題が混在する回は、1ファイル内で `ケース1 / ケース2 / ...` に分ける。
- 複数ケース運用でも、各ケースの条件・結論は混線させない。
- 「たまたま直った」ケースも残す（再現条件を失わないため）。

## 解説密度（固定）

- 短い要約だけで終えず、途中判断（どこで迷ったか / 何を除外したか）を残す。
- 多少雑多でもよいので、後から「なぜそう判断したか」を追える記録を優先する。
- 最終結論だけでなく、条件 / 理由 / 例外の3点を最低限残す。

## 本線反映のルール

- 本線/実務側への反映は「日時のケース補足を追加する」方式ではなく、既存カテゴリへ統合して一般化知識として残す。
- 既存カテゴリで収まらない場合は、2階層目以降で新規カテゴリ（フォルダ/ファイル/見出し）を追加してよい。
- ケース反映の要約は、反映先ではなくケースファイル末尾の「本線への反映先」に集約する。

## 記録テンプレ

```md
# NN_ケース_テーマ

## このページの扱い
- 単一議題 or 複数議題（ケース分割）

## ケース1_お題名
### 現象
- 何が崩れたか（見た目 / 挙動）

### 前提条件
- HTML/CSS条件
- JS条件

### 仮説
- どの条件が効いているか

### 検証ログ（1回に1条件）
1. 変更した条件
2. 出力
3. 判定

### 採用した形
- 最終実装
- 採用理由（条件 / 理由 / 例外）

### 見送った形
- 不採用理由

## ケース2_お題名（必要時のみ）
- 同じ形式で記録

## 本線への反映先
- どのカテゴリに何を戻したか
```

## 混在UI（JS×HTML/CSS）の扱い

- 主因を即断しづらい混在ケースは、このフォルダで管理する。
- 主因が明確な場合のみ、カテゴリ側ケースへ置く。
  - 見た目主因: HTML・CSS側
  - 挙動主因: JS側
- ただし「主ファイルは1本」に固定し、他側は参照リンクと要点だけ残す。

## ケース分析から本線へ戻す手順

1. ケース側で結論まで検証する
2. 再利用原理だけ抽出する
3. 抽出原理を該当カテゴリ本線に追記する
4. ケースファイル末尾の「本線への反映先」に反映結果を追記する
5. ケース本文は履歴として残し続ける
