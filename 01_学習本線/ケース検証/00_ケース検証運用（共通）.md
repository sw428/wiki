# 00_ケース検証運用（共通）

## 目的

- ここに貼られた内容を、まずケース検証ファイルとして形成して履歴化する。
- その後、再利用価値のある原理だけを各カテゴリ本線へ戻す。
- 「正解集」ではなく「条件 -> 出力 -> 採用理由」の検証ルートを残す。

## 基本フロー（固定）

1. まず `01_学習本線/ケース検証` に 1ケース1ファイルで記録する  
2. ケース内で現象/条件/出力/採用理由を確定する  
3. 各カテゴリ本線（HTML・CSS / JS / React / TypeScript）へ必要知識だけ反映する  
4. 各カテゴリの `メモ` に「ケース -> 本線反映」ログを残す

## ファイル命名

- `00`: 運用ルール専用（このファイル）
- `01` 以降: `NN_ケース_<短いケース名>.md`
- 例:
  - `01_ケース_Blogカード画像_レンダリング挙動.md`
  - `02_ケース_aria-hiddenとモーダル状態同期.md`

## 1ファイルに書く範囲

- 1ファイル1状況に限定する。
- 複数現象が混ざる場合は主問題ごとに分ける。
- 「たまたま直った」ケースも残す（再現条件を失わないため）。

## 記録テンプレ

```md
# NN_ケース_テーマ

## 現象
- 何が崩れたか（見た目 / 挙動）

## 前提条件
- HTML/CSS条件
- JS条件

## 仮説
- どの条件が効いているか

## 検証ログ（1回に1条件）
1. 変更した条件
2. 出力
3. 判定

## 採用した形
- 最終実装
- 採用理由

## 見送った形
- 不採用理由

## 本線への反映先
- どのカテゴリに何を戻したか
```

## 混在UI（JS×HTML/CSS）の扱い

- 主因を即断しづらい混在ケースは、このフォルダで管理する。
- 主因が明確な場合のみ、カテゴリ側ケースへ置く。
  - 見た目主因: HTML・CSS側
  - 挙動主因: JS側
- ただし「主ファイルは1本」に固定し、他側は参照リンクと要点だけ残す。

## ケース分析から本線へ戻す手順

1. ケース側で結論まで検証する
2. 再利用原理だけ抽出する
3. 抽出原理を該当カテゴリ本線に追記する
4. `メモ` に振り分けログを残す
5. ケース本文は履歴として残し続ける
