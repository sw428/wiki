# 00_BEM運用と命名判断

## 位置づけ（実務編レベル2）

このファイルは、実践でBEM命名を使うときの実務ルールを固定するノート。

- レベル1（基礎概念）: `01_基礎概念`
- レベル2: `02_実践`

## このノートの目的

- クラス名だけ見て、役割を追える状態にする
- 詳細度競争を避け、修正コストを下げる
- SCSS分割（components/pages）と責務を揃える

## BEMの基本

- Block: 部品のまとまり
  - 例: `.contact-form`, `.faq`, `.card`
- Element: Blockの内部要素
  - 例: `.contact-form__field`, `.faq__question`
- Modifier: 状態・差分
  - 例: `.button--primary`, `.faq__item--open`

## 命名ルール（案件で固定）

1. `__` は基本1段まで（深いDOMをそのまま名前にしない）
2. Modifierは「状態差分」に限定する（見た目微調整の乱用を避ける）
3. 同じ役割の繰り返し要素は同じElement名を使う
4. 作品名・文言依存でElement名を分けない

## やらないこと

- `block__element__element` のような多段命名
- ページ都合の余白調整をBlock本体へ直接埋め込む
- IDセレクタやタグ連鎖で強引に勝たせる

## Blockを分ける判断

### 分ける（別Blockにする）

- 単体で再利用する可能性がある
- ページが変わっても意味が崩れない
- 独立した状態管理を持つ

### 分けない（Elementのままにする）

- 親Blockがないと意味が成立しない
- その文脈内でのみ使う部品

## `contact-card` と `contact-form` の扱い（今回の最重要ポイント）

今回の案件では次で固定する。

- `contact-card`: フォームを包む外箱（背景、角丸、内側余白）
- `contact-form`: 入力・送信の本体
- `contact-form__fields`: 入力項目群
- `contact-form__field`: 1項目（label + input）

### 前提

BEMでは、Blockの中に別Blockを入れてよい。  
そのため `contact-card` の中に `contact-form` があって問題ない。

### 今回あえて分けた理由

同一コンポーネント内であっても、今回は責務を分離したほうが保守しやすいから。

- `contact-card` は「外箱の見た目」を担当する
- `contact-form` は「入力・送信という機能の構造」を担当する

この分離により、修正時の探索先が明確になる。

- 背景・角丸・paddingを直す: `contact-card` を見る
- 項目間隔・項目構造・送信周りを直す: `contact-form` を見る

### 迷いやすいポイント（今回の判断）

初期段階では、1つの入れ子グループを1 Blockで統一する運用は有効。

- 命名の負荷が低い
- 重複や命名ブレが出にくい
- BEM詳細よりCSSプロパティ理解に集中しやすい

一方で今回は、外箱とフォーム本体を同じBlockにまとめると責務が広がりやすい。  
そのため `contact-card__field` ではなく `contact-form__field` を採用し、要素の所属先を意味に合わせて分離する。

### この判断で得る実務メリット

- 命名から意図が読み取りやすくなる
- 自分で命名と設計を決める練習になる
- チームで一貫した読み方を作りやすい
- 他人のコードを読むときも責務境界を追いやすい

補足として、セレクタ名がやや長くなること自体は、通常規模の案件で性能ボトルネックになりにくい。  
このため今回は、短さより責務の明確さを優先する。

### GPTでBEM初期案を作る運用

初期段階では、1つの入れ子グループに対して  
「1 Block + 必要な Element」をGPTに出してもらってよい。

- 狙い:
  - 命名の負荷を下げる
  - 重複や命名ブレを減らす
  - 詳細度やセレクタ設計の混線を減らし、CSSプロパティ検証に集中する
- 注意:
  - GPT案は初期案として使い、最終決定は責務境界で再判定する
  - 見た目責務と機能責務が混ざる場合は、今回のようにBlockを分離する

実務の流れは次で固定する。

1. GPTで初期BEM案を作る
2. 人間が責務（条件 / 理由 / 例外）で再判定する
3. 必要な箇所だけBlockを再分割して確定する

#### GPT依頼テンプレ（初期案）

```txt
次のHTML構造に対して、BEM命名の初期案を作ってください。

条件:
- 1入れ子グループにつき、基本は1Blockで開始
- Elementは `__` 1段まで
- Modifierは状態差分のみ

出力:
1. class一覧
2. Block / Element 対応表
3. 責務が混ざって見える箇所（分割候補）
```

## SCSS分割との対応

- `components`: 部品の見た目（Block側）
- `pages`: 配置・余白・並び（文脈側）

`pages` 側で `components` の核を上書きし続けると、BEMを使っても保守性は下がる。
命名だけでなく、責務配置もセットで運用する。
