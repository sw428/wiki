# 10_ケース_ケース入口と反映先振り分け判断

## 1. 事実（運用文脈）

- 前回整理では、ケース検証で受けたあとに基礎概念へ戻す流れは作ったが、「まず2つのケース検証入口に分ける」という前提が弱かった。
- 混在ログでは、次の話題が同時に入ることがある。
  - 特定ポートフォリオの実装判断
  - HTML/CSSの基本原理
  - BEM命名やSCSS分割のような実務汎用ルール
- そのため、反映先を先に決めるのではなく、ケース検証で受けてから振り分ける流れを固定する。

## 2. 判断プロセス

### ケース1: 特定ポートフォリオの話

#### 判定

- 入口: `02_実践/ケース検証`
- 反映先:
  - 案件固有の実践判断 -> `02_実践/ポートフォリオサイト/<案件名>/...`
  - 抽象化できるHTML/CSS原理 -> `01_基礎概念/HTML・CSS`

#### 理由

特定サイトの構造、素材、カンプ、命名、ページ構成に依存する判断は、まず実践ケースとして受ける。
そこから、案件名や素材条件を外しても成立する原理だけを基礎概念へ戻す。

#### 例

- マリンプラクティスの main / aside 2カラム
- マリンプラクティスのCTAボタン
- マリンプラクティスのVoiceカード配置

### ケース2: 一般的なBEM / SCSSの話

#### 判定

- 入口: `02_実践/ケース検証`
- 反映先:
  - BEM命名の実務判断 -> `02_実践/BEM運用/00_BEM運用と命名判断.md`
  - SCSS導入・分割運用 -> `02_実践/SCSS導入/00_SCSS導入と分割管理.md`
  - 基礎概念として再利用できるCSS原理 -> 必要に応じて `01_基礎概念/HTML・CSS`

#### 理由

BEMやSCSSは、純粋な基礎概念というより、実践で使う実務運用として扱う場面が多い。
そのため、基礎概念へ自動で入れず、まずケースで「一般運用なのか、特定案件固有なのか」を分ける。

#### 例

- BEMのBlock / Elementをどこで切るか
- 画像やアイコンへクラスを付けるか
- SCSSの `components` / `pages` 分割
- `@use` の順番と詳細度の混同

## 3. 採用する流れ

1. 貼り付けログを読む
2. まずケース検証入口を選ぶ
   - 学習寄り -> `01_基礎概念/ケース検証`
   - 案件・実務寄り -> `02_実践/ケース検証`
3. 混在している場合は、必要に応じてケースを2つに分ける
4. ケース本文で、特定条件 / 一般原理 / 実務運用を分ける
5. その後に反映先を決める
   - 基礎概念
   - 特定案件フォルダ
   - BEM運用
   - SCSS導入
   - 反映せずケースだけ残す

## 4. 基礎概念へ戻すもの（01側）

- HTML/CSS/JSなど、案件名を外しても成立する基本原理
- 表示ボックス、行ボックス、Flex/Grid、画像比率などの確認順
- BEM/SCSSの話でも、CSSの基本原理として切り出せる部分

## 5. 実践へ戻すもの（02側）

- 特定ポートフォリオの実践判断
- BEM命名やSCSS分割のような実務運用
- 案件ごとの既存構造、命名方針、素材条件に依存する判断

## 6. 反映先判定

- 反映先:
  - `02_実践/ケース検証/00_実践ケース運用（共通）.md`
  - `01_基礎概念/ケース検証/00_ケース検証運用（共通）.md`
- 判定理由:
  - 今回は個別プロパティの知識追加ではなく、ケース入口と反映先を決める運用ルールの整理であるため。
- 反映内容（要約）:
  - まず2つのケース検証入口で受ける。
  - その後、特定ポートフォリオ / 基礎概念 / BEM運用 / SCSS導入へ振り分ける。
  - 混在ログは、必要に応じてケースを2つに分けてから反映する。
