## ケース2_aspect-ratio と width責務の切り分け

### 現象

- `aspect-ratio` を指定しているのにSPでカードや画像枠が想定より大きくなる。

### 前提条件

- 親要素が `max-width: none` や `width: 100%` で広がる構成。
- 画像は `width:100%; height:100%; object-fit:cover` で枠充填。

### 仮説

- `aspect-ratio` は比率を決めるだけで、幅の上限は止めない。
- 枠の実寸は `width / max-width / grid列幅` 側で決まる。

### 検証ログ（1回に1条件）

1. `aspect-ratio` のみ指定（幅が広がると高さも連動して拡大）
2. 親または枠に `width: min(100%, 329px)` を追加
3. 判定: 比率指定とサイズ上限指定を分離すると挙動が安定

### 採用した形

- 比率: `aspect-ratio`
- サイズ上限: `width` / `max-width`
- 枠充填: `img { width:100%; height:100%; object-fit:cover; }`

### 見送った形

- `aspect-ratio` だけで実寸上限まで制御しようとする設計

## ケース3_absoluteピン配置（地図とのズレ）

### 現象

- `position:absolute` を付けたピンが「左上に行く想定」と違う位置になる。
- PC/SPで地図とピンの対応位置がズレる。

### 前提条件

- 親: `.access-map { position: relative; }`
- ピン: `.access-map__pin { position: absolute; }`
- 地図: `object-fit: cover` で表示範囲がブレイクポイントごとに変わる。

### 仮説

- `absolute` だけでは左上固定にならず、`top/left` 指定が必要。
- ピンは「地図座標」ではなく「親枠座標」に置かれるため、表示範囲が変わるとズレて見える。

### 検証ログ（1回に1条件）

1. `position:absolute` のみ（期待位置にならない）
2. `top:0; left:0;` を追加（左上固定を確認）
3. `left/top` をSP/PCで分けて調整
4. `object-position` をSP/PCで微調整
5. 判定: 同一画像運用なら、地図側表示範囲調整 + ピン位置調整のセットが必要

### 採用した形

- 地図画像は基本1枚を共通利用。
- ピンは別要素で absolute 配置。
- SP/PCで `object-position` とピン座標を分岐して合わせる。

### 見送った形

- `position:absolute` のみで自動的に意図位置へ来る前提
- 実住所座標の厳密性を優先してカンプ見た目を崩す実装

## ケース4_Contact情報カードのセマンティック判断

### 現象

- `div / section / address / ul` のどれを使うか判断が揺れる。

### 前提条件

- 見出しを持つ独立セクションではなく、Contact領域内の情報カード。
- Phone / Email / Address の同種項目が並ぶ。

### 仮説

- 外側は「章」ではなく配置用ブロックなので `div` が自然。
- 中身は連絡先情報なので `address` が自然。
- 項目は同種反復なので `ul > li` が自然。

### 検証ログ（1回に1条件）

1. `section` 案（見出し不在だと章としては過剰）
2. `div + address` 案（意味と見た目責務の分離が明確）
3. 項目反復を `ul/li` 化
4. 判定: `div > address > ul > li` が最も読みやすい

### 採用した形

- `div.contact-info`（外枠）
- `address.contact-info__body`（連絡先意味）
- `ul.contact-info__list > li.contact-info__item`（反復項目）

### 見送った形

- 見出しなしで `section` を多用する構造
- 反復項目をすべて `div` で並べるだけの構造

## ケース5_カード共通化の判定軸（形ではなく役割）

### 現象

- 「四角いカードなら同一コンポーネント化すべきか」で判断が揺れる。

### 前提条件

- 見た目が似ていても、カードの役割・中身が異なる場合がある。

### 仮説

- 共通化基準は形状ではなく、役割・骨組み・再利用範囲。

### 検証ログ（1回に1条件）

1. 見た目中心の共通化案を検討
2. 役割/構造中心で再分類
3. 判定: 差分がサイズ・余白程度なら共通化、役割差が大きいなら分離

### 採用した形

- 「見た目でまとめない、構造でまとめる」を運用基準化。

### 見送った形

- 四角形であることだけを理由にした共通化

## どういうことが分かったか

- レイアウト崩れは「画像サイズ」より先に「枠の幅決定元」を見ると速い。
- `aspect-ratio` / `width` / `object-fit` の責務分離で、再発時の切り分けが安定する。
- 地図ピンは地図座標固定ではなく親枠固定であることを前提に、SP/PCで調整点を持つ必要がある。
- セマンティックは「見た目」ではなく「章か/連絡先か/反復か」の役割分解で判断すると迷いにくい。
