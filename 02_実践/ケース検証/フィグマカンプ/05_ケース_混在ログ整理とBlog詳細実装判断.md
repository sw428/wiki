# 05_ケース_混在ログ整理とBlog詳細実装判断

## 記録日

- 2026-05-05

## このページの扱い

- 複数議題（ケース分割）
- 1つの貼り付けログに、医療相談・CSS実装・BEM命名・表示制御・エディタ運用が混在していたため、フロントエンド実装に関係する部分だけをケース化した。

## ケース1_受領ログのスコープ分離

### 現象

- 1つの貼り付け内に「体調/薬の相談」と「フロント実装相談」が同時に含まれていた。

### 前提条件

- この repo はフロントエンド知識整理を主目的とする。
- 健康面の個人情報や私生活の個人話題は記録対象外。

### 仮説

- すべてを同じ粒度で保存すると、repo の目的（再利用できるフロント知識）から外れ、後で検索導線が崩れる。

### 検証ログ（1回に1条件）

1. 受領ログを話題で分割（医療 / 実装 / 命名 / レイアウト）
2. 実装再現に必要な話題だけ抽出
3. 判定: フロント知識に直接関係しない体調・薬の詳細は保存しない方が運用方針に一致

### 採用した形

- フロント実装の判断材料だけをケースとして保存。
- スコープ外の個人話題は「除外理由のみ」残し、内容自体は残さない。

### 見送った形

- 受領ログ全文の転載保存（理由: repo の対象範囲を超えるため）

## ケース2_authorブロック中央寄せ（`text-align` とリスト整列の分離）

### 現象

- 著者名は中央に見えるが、SNSリンクだけ左に残る。

### 前提条件

- 親要素に `text-align: center;`
- SNS部分は `ul > li > a` のリスト構造

### 仮説

- `text-align` は主にインライン内容に効くが、`ul` はブロック要素なので、その中身を横中央にするには別の整列文脈が必要。

### 検証ログ（1回に1条件）

1. 親だけ `text-align: center` を適用
2. `ul` に `display: flex; justify-content: center;` を適用
3. 判定: 文字中央寄せとリスト中央寄せを分離すると安定

### 採用した形

- 親: `text-align: center`
- SNSリスト: `display: flex; justify-content: center; gap: ...`
- 画像: `display: block; margin: 0 auto;` で中央固定

### 見送った形

- `text-align` だけで `ul` の整列まで解決しようとする実装

## ケース3_共通化の境界（`c-` とページ固有クラス）

### 現象

- `blog-card` を複数ページで使い始めた段階で、`c-` を付ける範囲とページ固有の調整範囲が混線。

### 前提条件

- カード本体は再利用される。
- `grid-column` や大型カード比率はページのグリッド事情で変わる。

### 仮説

- 見た目の核とページ配置を同じクラスに混ぜると、共通化後に他ページへ副作用が出る。

### 検証ログ（1回に1条件）

1. カード本体を `c-blog-card` に統一
2. 大型配置は `blog-card--large` をページ側SCSSで管理
3. 判定: コンポーネント核とページ配置の責務分離が最も読みやすい

### 採用した形

- `components`: `c-blog-card` 本体（メディア枠・本文パーツ）
- `pages`: `blog-card--large` の列幅/比率/タイトルサイズ調整

### 見送った形

- ページ都合の `grid-column` まで `c-` 側へ寄せる構成

## ケース4_inline文脈での画像下隙間（`inline-flex` / `flex` / `block`）

### 現象

- `margin` を付けていないのに、画像リンク下に余白が見える。
- `inline-flex` だと残り、`flex`/`block` にすると詰まる。

### 前提条件

- `a` が `inline-flex` or `inline-block`
- 画像はリンク内に1枚

### 仮説

- 原因は margin ではなく、inline formatting context の baseline/descender 由来。
- `inline-flex` は「内側はflex」でも「外側はinline」のため、行ボックス都合が残る。

### 検証ログ（1回に1条件）

1. `a` を `inline-flex` に設定（下隙間が残る）
2. `a` を `flex` / `block` に変更（隙間が消える）
3. `img` を `display: block; width:100%; height:100%; object-fit:cover;` に固定
4. 判定: 画像枠では inline 系を避ける方が再現性が高い

### 採用した形

- 画像リンクは `display: block`
- 画像は `display: block` + `width/height: 100%` + `object-fit: cover`

### 見送った形

- `inline-flex` 維持のまま都度 `vertical-align` 調整で逃げる運用

## ケース5_SP/PC表示差分と `time` の意味整合

### 現象

- SP表示とPC表示で日付文字列を変える案が出たが、同一記事カードとして意味がぶれる。

### 前提条件

- `time` 要素の `datetime` は1要素に1値
- 同一記事なら日付データは本来1つ

### 仮説

- カンプ差分をそのまま実装すると、見た目は合っても意味データが矛盾しやすい。

### 検証ログ（1回に1条件）

1. SP/PCで別日付を出し分ける案を確認
2. `datetime` との整合を確認
3. 判定: 同一記事なら日付を統一し、必要ならコメントでカンプ差分理由を残す

### 採用した形

- 優先順位は「データの正しさ > カンプ上の文字差」
- 同一記事カードは日付を統一
- カンプ差分は短い実装コメントで明示

### 見送った形

- 同一記事なのにSP/PCで異なる日付を持たせる実装

## ケース6_`l-inner` と `l-narrow` の責務分離（大画像の扱い）

### 現象

- ブログ詳細のメイン画像を大きくしたいが、`l-narrow` 内に入っていて幅が広がらない。

### 前提条件

- `l-narrow` は本文向けの狭幅コンテナ
- 画像はヒーロー的に広幅表示したい

### 仮説

- 本文幅コンテナとメディア幅コンテナを分離すれば、無理なはみ出し調整なしで意図どおりの幅になる。

### 検証ログ（1回に1条件）

1. 画像を `l-narrow` 内に残す（狭幅のまま）
2. 画像を `l-inner` 直下へ移動
3. 判定: 見出し/本文は `l-narrow`、メイン画像は `l-inner` の分離が最も自然

### 採用した形

- `header/body`: `l-narrow`
- `figure`: `l-inner` 直下 + 必要なら `max-width` 指定

### 見送った形

- `l-narrow` 内に置いたまま transform で無理に広げる実装

## どういうことが分かったか

- 混在ログは「話題分類 -> スコープ判定 -> 実装原理抽出」の順で処理すると崩れにくい。
- 画像/リンクのズレは `margin` 疑いから入るより、まず formatting context（inlineかblockか）を確認した方が早い。
- `c-` 共通化は「再利用核」に限定し、配置都合はページ側へ残す方が保守しやすい。
- カンプ差分があっても、`time` など意味データはシステムの正規値を優先した方が後工程で安全。
