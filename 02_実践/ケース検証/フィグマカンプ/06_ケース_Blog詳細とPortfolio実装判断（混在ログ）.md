# 06_ケース_Blog詳細とPortfolio実装判断（混在ログ）

## 記録日

- 2026-05-06

## このページの扱い

- 複数議題（ケース分割）
- 1つの貼り付けログに、体調・薬の相談、HTML/CSS実装、BEM、DevTools運用、VS Code操作が混在していたため、フロントエンド実装へ再利用できる部分だけを抽出した。

## ケース1_受領ログのスコープ分離

### 現象

- 医療相談と実装相談が同一ログに混在していた。

### 前提条件

- この repo はフロントエンド知識整理が主目的。
- 健康面に関する個人情報や私生活の個人話題は記録対象外。

### 仮説

- 全文保存すると検索導線が崩れ、再利用性が下がる。

### 検証ログ（1回に1条件）

1. 受領ログを「医療」「実装」「運用」に分解
2. 実装再現に必要な話題のみ抽出
3. 判定: 除外ルールを明示して実装知識だけ保存する方が運用方針に一致

### 採用した形

- フロント実装に直結する判断だけをケース化。
- スコープ外内容は「除外理由のみ」残し、本文は残さない。

### 見送った形

- 受領ログ全文の転載

## ケース2_author領域の中央寄せとBEM整理

### 現象

- `text-align: center` を入れても、SNSリストや名前位置が意図どおり中央に見えない場面があった。

### 前提条件

- 著者領域は `img`、名前/役職、SNSリストで構成。
- 途中で `div` の役割名が不足していた。

### 仮説

- 親の `text-align` とリスト整列は責務が別。
- `align-items: center` は「要素箱同士」の中央揃えであり、名前1行だけは中央にならない。
- SP用に残った `margin` がPC見え方を崩している。

### 検証ログ（1回に1条件）

1. 親だけ `text-align: center` を適用
2. `ul` に `display: flex; justify-content: center` を追加
3. 著者名ズレ箇所を DevTools で確認すると、SP時の `margin` が残存
4. 判定: BEMの箱分割と余白責務分離で安定

### 採用した形

- `blog-detail__author-inner`、`blog-detail__author-profile`、`blog-detail__author-text` を付与。
- SNSは `blog-detail__social-list` 側で `flex` 整列。
- 名前/役職の余白は個別 `margin` を減らし、親側の `gap` 管理を優先。

### 見送った形

- `text-align` だけで全要素整列を解決しようとする実装

## ケース3_関連記事カードの共通化境界

### 現象

- `blog-card` を複数ページで使う中で、`c-` へ寄せる範囲とページ固有スタイルの範囲が混線した。

### 前提条件

- カード本体は再利用する。
- `grid-column` や `--large` はページグリッド都合で変わる。

### 仮説

- コンポーネント核とページ配置を分けると、他ページ副作用を減らせる。

### 検証ログ（1回に1条件）

1. カード本体を `c-blog-card` 系へ統一
2. 配置調整は `blog-card--large` としてページSCSSに残す
3. 判定: 役割分離のほうが実装意図を追いやすい

### 採用した形

- 共通部品: `c-blog-card`（components）
- ページ配置: `blog-card--large`（pages）
- `section + ul/li + article` は一覧意図が強い時に採用、既存整合優先時は `div + article` を許容

### 見送った形

- ページ都合のグリッド指定まで `c-` 側へ集約する構成

## ケース4_inline文脈の下隙間（`inline-flex` と baseline）

### 現象

- 画像リンク下に `margin` なしでも隙間が残る。
- `inline-flex` では残り、`flex` や `block` にすると消える。

### 前提条件

- `a` の中に画像1枚の構造。
- `aspect-ratio` を親に持つ画像枠。

### 仮説

- 原因は `margin` ではなく、inline formatting context の baseline/descender。
- `inline-flex` は内側flexでも外側inlineなので行ボックス影響が残る。

### 検証ログ（1回に1条件）

1. `a` を `inline-flex` に設定（隙間が残る）
2. `a` を `flex` / `block` に変更（隙間が解消）
3. 画像を `display:block; width:100%; height:100%; object-fit:cover` に固定
4. 判定: 画像1枚リンクは `block` が最も再現性が高い

### 採用した形

- 画像リンクは `display: block`
- 枠充填は `height:100% + object-fit:cover` をセット運用

### 見送った形

- `inline-flex` のまま `vertical-align` 微調整で逃げる運用

## ケース5_`@media` の閉じ忘れと DevTools読解順

### 現象

- 「PC条件を書いたのにSPで効く」状態が発生した。

### 前提条件

- `@media (min-width: 768px)` の閉じ位置が混在。
- DevToolsで子要素を選択していたため、親の `display:flex` を見落としやすい。

### 仮説

- 早い位置で `}` が閉じ、意図したルールが `@media` 外へ漏れた。

### 検証ログ（1回に1条件）

1. `@media` ブロックの閉じ位置を確認
2. `Computed` で `display` の最終値とソース行を確認
3. 親 `ul` を選択し、子 `li` ではなく container 側の指定を確認
4. 判定: `Styles` と `Computed` の確認順を固定すると再発防止しやすい

### 採用した形

- PC専用指定は `@media` ブロック内にまとめる。
- デバッグ時は「要素選択 -> Computed -> 値の出どころ」の順で確認する。

### 見送った形

- 子要素だけ見て、親のレイアウト指定を推測で判断する進め方

## ケース6_aspect-ratio・2倍書き出し・予約枠の切り分け

### 現象

- `aspect-ratio` を入れても想定より大きくなるケースがあった。
- 2倍書き出し画像の `width`/`height` を何で書くか混乱した。

### 前提条件

- 画像枠は `aspect-ratio` で固定。
- 親側に `width` / `max-width` / グリッド幅指定が存在する。

### 仮説

- `aspect-ratio` は比率指定であり、最大値制限ではない。
- 実サイズは親幅で決まり、画像は `width/height:100%` で追従する。
- 2倍書き出しは「表示寸法での設計」と分離して扱うべき。

### 検証ログ（1回に1条件）

1. `aspect-ratio` のみ指定（親幅次第で拡大）
2. `max-width` を調整（意図どおり停止）
3. 2倍画像の表示寸法を1/2換算して設計
4. 判定: 比率指定とサイズ上限指定は別責務として固定

### 採用した形

- 比率: `aspect-ratio`
- サイズ上限: `width` / `max-width` / grid列幅
- 画像埋め: `width:100%; height:100%; object-fit:cover`
- 2倍画像は表示寸法を基準に属性・設計値を決定

### 見送った形

- `aspect-ratio` だけで実寸上限まで制御できる前提

## ケース7_意味データ優先（`time` / `figure`）

### 現象

- SP/PCで日付文字列が違うカンプ差分があり、`datetime` 整合が崩れそうになった。
- メイン画像で `figure` を使うか `div` を使うか判断が揺れた。

### 前提条件

- 同一記事カードなら日付データは1つ。
- `figure` は独立図版や `figcaption` 付き文脈で強い。

### 仮説

- 見た目差よりデータ整合を優先した方が後工程で安全。

### 検証ログ（1回に1条件）

1. SP/PCで別日付表示案を確認
2. `datetime` の意味整合を確認
3. `figure` 必須条件（独立図版/キャプション有無）を確認
4. 判定: 同一記事日付は統一し、必要ならコメントで差分理由を残す

### 採用した形

- 優先順位を「意味データの正しさ > カンプ差分」に固定。
- キャプションなしのメイン画像は `div` でも可、図版として扱うなら `figure` を使う。

### 見送った形

- 同一カードでSP/PCの日付を別データで保持する実装

## どういうことが分かったか

- 混在ログは、先にスコープ分離しないと知識化の品質が落ちる。
- レイアウト崩れは「margin」より先に「どの formatting context にいるか」を確認した方が速い。
- `aspect-ratio` は形、`width/max-width/grid` は大きさ、`object-fit` は中身の見せ方、の3分離が有効。
- DevToolsは `Styles` 単独より `Computed` で最終値の出どころ確認まで行う方が再現性が高い。
