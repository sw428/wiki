# 01_ケース_Blogカード画像_レンダリング挙動

## 目的

- 「正解の型」を暗記する前に、同じ条件なら同じ出力になることを言語化する。
- 途中挙動を追えるようにし、エラー時に「どこが変数だったか」を切り分けられる状態を作る。

## 対象層と観測視点

- 対象層: 実務初学〜中級（HTML/CSSで表示崩れを自力で追いたい段階）
- 観測視点:
  - 構造: 親子関係、レイアウトモード、命名責務
  - 見た目: 幅/高さ、切り抜き、余白、ズレ
  - 実装: `display` / `width` / `height` / `max-width` / `aspect-ratio` / `object-fit`
  - 運用: どの層に何を持つか（共通ルールと文脈差）

## 使い方

1. 現象を書く（例: 「画像だけ小さい」「下に隙間が出る」）
2. このページで「一致する条件」を探す
3. 予測される出力を先に言語化する
4. DevToolsで値を照合する
5. 条件を1つだけ変えて再確認する

## 1. 箱と幅の法則（Normal Flow）

### 1-1. `display: block` + `width: auto`

- 条件: ブロック要素、幅未指定
- 出力: 横幅は包含ブロックの利用可能幅まで広がる
- 関与: 通常フロー、包含ブロック

### 1-2. `display: block` + `height: auto`

- 条件: ブロック要素、高さ未指定
- 出力: 高さは中身（子要素）の合計で決まる
- 関与: 通常フロー、コンテンツベースの高さ

### 1-3. `max-width` は「その幅にする」ではない

- 条件: `max-width: 329px`
- 出力: 最終幅は「その時点の候補幅」と `329px` の小さい方
- 関与: used value 計算（`width` と `max-width` の競合）

### 1-4. `width: 100%` の基準

- 条件: 子要素に `width: 100%`
- 出力: ビューポートではなく「親の content box 幅」に対する 100%
- 関与: パーセント解決の基準は親

### 1-5. `max-width` と `padding` を同じ箱に置いた時

- 条件A: `box-sizing: border-box`
  - 出力: 指定幅の中に `padding` が含まれるため、内容幅は小さくなる
- 条件B: `box-sizing: content-box`（初期値）
  - 出力: 内容幅に `padding` が外付けされ、外寸が大きくなる
- 関与: box model の計算基準

### 1-6. 外側の箱と内側の箱の順序

- 条件: `l-inner`（共通余白）と `blog-articles__inner`（セクション固有幅）の順が入れ替わる
- 出力:
  - `l-inner` を外に置く: 先に共通余白を確保し、その内側で固有幅を制御
  - 固有幅を外に置く: 先に狭めた箱の中に共通余白が入り、実効幅がさらに減ることがある
- 関与: どの層で幅制限をかけたか（責務順）

### 1-7. 同一要素に複数クラスを付けた時

- 条件: `<div class="l-inner blog-articles__inner">`
- 出力: 2つの役割が1つの箱に合成され、`max-width` / `padding` / `margin` が同時に衝突する
- 関与: 多重クラスの合成結果（Cascade + Box Model）

## 2. 画像（置換要素）の法則

### 2-1. `img { width: 100%; height: auto; }`

- 条件: 画像は親幅に合わせ、縦は `auto`
- 出力: 高さは「表示幅 ÷ 元画像比」で決まる
- 例: 表示幅 `283px`、元比率 `329:224` -> 高さは約 `192.7px`
- 関与: 置換要素の intrinsic ratio

### 2-2. 親に `aspect-ratio`、子画像は `height: auto`

- 条件: 親枠の比率は固定、子は `height: auto`
- 出力: 親の高さは固定されても、子画像は元比率優先で高さが不足することがある
- 見え方: 枠に対して画像が伸び切らない
- 関与: 親の preferred ratio と子の intrinsic ratio の別計算

### 2-3. 枠いっぱいにしたい場合

- 条件: 子画像を `width: 100%; height: 100%; object-fit: cover;`
- 出力: 子画像は親枠を完全充填し、はみ出し部分は切り抜かれる
- 関与: `object-fit` は幅と高さが確定した箱で効く

### 2-4. `object-fit: cover` / `contain`

- `cover`: 枠は埋まる、どこかが切れる
- `contain`: 全体は見える、余白が出る
- どちらも「正誤」ではなく、条件（切り取り許容か）で選ぶ

### 2-5. `object-position` の方向

- 条件: `object-fit: cover` 前提
- 出力:
  - 1つ目（X）を小さく -> 右側が減りやすい
  - 2つ目（Y）を小さく -> 下側が減りやすい
- 例: `35% 35%` は中央（`50% 50%`）より左上寄り

### 2-6. HTML属性 `width` / `height` の意味

- 条件: `<img width="329" height="224">`
- 出力: CSS直書きでなくても、比率予約の計算情報として扱われる
- DevTools: `img[属性スタイル]` として見えることがある
- 関与: レイアウト予約（CLS抑制）

### 2-7. DevToolsで灰色の「属性スタイル」

- 条件: 属性由来値より強いCSSが後から適用
- 出力: 属性由来の `width/height` は打ち消し表示されるが、予約情報は残ることがある
- 関与: Cascade と計算結果表示

## 3. `display` と整列モードで出力が変わる

### 3-1. 画像下の謎の隙間（ベースライン）

- 条件: `img` が `display: inline` 文脈で描画
- 出力: テキストベースライン都合で下に隙間が見えることがある
- 対処: `img { display: block; }`
- 関与: インライン整形コンテキスト

### 3-2. `inline-flex` で急に揃って見える理由

- 条件: 親を `inline-flex`、`align-items` 既定値（`stretch`）
- 出力: 子が flex item となり、cross-axis 方向に引き伸ばされ、結果として高さが揃って見える
- 重要: 「画像比率を理解して直した」のではなく、「flex整列ルールが効いた」結果
- 関与: Flex formatting context、`align-items: stretch`

### 3-3. `inline-flex` は画像専用の正攻法ではない

- 条件: 後で `align-items: center/start` へ変更
- 出力: 伸び方が変わり、以前の見え方が崩れることがある
- 推奨: 画像枠は `block + aspect-ratio + img height:100% + object-fit` を基準に固定

### 3-4. `inline-flex` が「一発できれい」に見える時の内訳

- 条件: `a` を `inline-flex` にする
- 出力:
  - `a` 自体は inline-level で縮み方向に働きやすい（見た目が締まりやすい）
  - 子は flex item になり、既定の `align-items: stretch` で高さが揃って見えやすい
  - ベースライン隙間問題や高さ不足が同時に目立たなくなることがある
- 重要: 複数要因を同時に解消できる反面、「どの条件が効いたのか」が見えにくくなる

### 3-5. `stretch` は「明示していなくても効く」前提値

- 条件: 親が flex formatting context（`display: flex` / `inline-flex`）
- 出力: `align-items` 未指定でも既定値 `stretch` が適用される
- 注意: DevTools上で最初から目立って表示されないことがあり、見落とすと原因追跡を誤る
- 実務メモ: なんとなく `inline-flex` を使うと、後で調整が必要になった時に「親を触るべきか、子を触るべきか」が判断しづらくなる

### 3-6. `a { inline-flex }` を定型化しすぎない

- 使いどころ: ボタン、アイコン付きリンク、マーク類の整列（中身を揃える目的）
- 注意: 今回うまくいったからといって、画像枠にも同じ理由で常用すると後で崩れ原因を追いにくくなる
- 判断基準: 「何を揃えたいのか（テキスト/アイコン整列か、画像トリミングか）」を先に分ける
- 実務メモ: 画像枠は原則 `block + aspect-ratio + img 100%/100% + object-fit` を先に採用し、`inline-flex` は整列目的で必要な時だけ使う

## 4. BEMと責務分離の出力パターン

### 4-1. セクション骨格名を途中で変える

- 条件: 同じ骨格レイヤーで親ブロック名が揺れる
- 出力: 責務追跡が難しくなり、修正時に親子関係の再把握コストが増える

### 4-2. `block__element` を本体なしで開始

- 条件: `newsletter__inner` だけ先に出す
- 出力: 所属ブロックが不在になり、命名が読み手にとって解釈依存になる

### 4-3. 独立ブロックの配置パターン

- パターンA（明示型）:
  - `blog-articles__newsletter > newsletter`
  - 出力: 配置責務と部品責務が分離され、読みやすい
- パターンB（Mix型）:
  - `newsletter blog-articles__newsletter`
  - 出力: DOMは薄くなるが、1要素に複数責務が合成される

### 4-4. `c-` 接頭辞の運用

- 条件: 他ページ再利用が確定（目安3回以上）
- 出力: `c-newsletter` など共通部品として管理しやすくなる
- 条件: 文脈依存が強い
- 出力: `newsletter` / `blog-card` のままの方が意図が読みやすい

---

## 5. デバッグ手順（条件1つずつ）

1. 親から見る（幅・`display`・`box-sizing`・`padding`）
2. 子を見る（`width` / `height` / `max-width` / `object-fit`）
3. 画像なら intrinsic ratio を確認する
4. DevToolsで最終採用値（Computed）を確認する
5. 1回に1条件だけ変えて差分を取る
6. GPT提案に違和感がある時は、「同じタグでも前提条件が何か」を分解して確認してから採用する

---

## 6. 最終形（ベストプラクティスは最後に固定）

```css
.media-frame {
  display: block;
  aspect-ratio: 329 / 224;
  overflow: hidden;
}

.media-frame > img {
  display: block;
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

- ここだけ覚えるのではなく、「なぜこれで揃うか」を上の条件表で逆引きできる状態を維持する。
