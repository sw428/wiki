# 05_ケース_見出し階層とfooter構造判断

## 1. 事実（案件文脈）

- 対象は、HTML断片レビュー時の構造相談。
- 相談の主論点:
  - `h4` が先に出てくる構造は許容か
  - その要素を `p` / `h2` のどちらへ寄せるべきか
  - `footer` をどう置くべきか（ページ全体と部分領域）
- 同一ログ内にキャリア/市場系の話題が混在していたが、本ケースはフロント実装構造のみ扱う。

## 2. 判断プロセス

### ケース1: 先頭に `h4` が来る場合

#### 候補A: 見た目サイズ基準で `h4` を維持する

- 判定: 不採用
- 理由: 見出しタグは見た目サイズではなく文書階層で決める。

#### 候補B: 階層上の親見出しがないなら `h2` / `h3` へ引き上げる

- 判定: 採用（既定）
- 理由: 見出しレベルの飛び級（`h2`/`h3` 不在で `h4`）は読み上げ導線を崩しやすい。

#### 候補C: 見出しではなくラベル文なら `p` / `span` に落とす

- 判定: 採用（条件付き）
- 理由: 章立てを作らない文言は見出しタグにしない方が構造が明確。

### ケース2: `h2` と `p` の分岐

- `h2` を使う条件:
  - その要素が「新しい章/セクションの主題」を表す
  - 下位に本文やサブ要素が続く
- `p` を使う条件:
  - 章見出しではなく、本文導入や補足文である
  - 下位階層を作る意図がない

### ケース3: `footer` の置き方

#### 候補A: ページ末尾の共通情報を1つの `footer` に集約

- 判定: 採用（既定）
- 理由: サイト全体のフッターランドマークとして扱いやすい。

#### 候補B: 各 `article` / `section` にローカル `footer` を置く

- 判定: 条件付き採用
- 理由: 投稿日時、タグ、補助リンクなど「その領域専用の締め情報」なら意味がある。

#### 候補C: すべて `div` で代用

- 判定: 不採用
- 理由: フッター領域の意味とナビゲーション補助が弱くなる。

## 3. 案件固有として残すもの（02側）

- `Pasted code(35).html` / `Pasted code(36).html` の詳細断片が未展開でも、構造判断は次の順で先に固定できる。
  1. その文言は章題か本文か
  2. 章題ならどの階層か（親見出しとの関係）
  3. フッター情報は全体用か局所用か

## 4. 基礎概念へ戻すもの（01側）

- 再利用可能な原理:
  - 見出しタグは見た目ではなく「文書アウトライン」で決める。
  - 飛び級見出し（例: 親不在の `h4`）は原則避ける。
  - ページ全体の締めはグローバル `footer`、領域単位の締めはローカル `footer` を使い分ける。
- 判断順:
  1. そのテキストは章題か本文か
  2. 章題なら `h1` からの階層連続性を確認
  3. フッター情報のスコープ（全体/局所）を確定
  4. 見た目調整はCSSで行い、タグ選択と分離する
- 例外条件:
  - CMS由来で見出しレベルが固定される場合は、上位構造側で連続性を補正する。
