# 05_ブラウザ挙動とズレ

## 目的

- 「なぜ画面がずれるか」を、レンダリング工程の因果で説明できるようにする。

## 基本フロー

`HTML解析 → DOM → CSSOM → Render Tree → Layout → Paint → Composite`

- この一連をレンダリングとして扱う。
- [DOM](../../03_雑記/知識辞書（試運用）/02_単語カード/DOM.md) と [CSSOM](../../03_雑記/知識辞書（試運用）/02_単語カード/CSSOM.md) は、構造とスタイル指定を分けて見るための入口。
- `Layout` は位置とサイズの計算工程。

補足:
- 画面に見える時点では結果がかなり出ている
- ただし内部では、計算順序と依存関係に沿って確定している

## 用語の関係

- Reflow: レイアウト再計算
- Layout Shift: 再計算の結果、表示位置が動く現象
- CLS: Layout Shift の累積指標
- Scroll Anchoring: Shift時に閲覧位置を保とうとするブラウザ補正

## 今回の主因パターン

`レスポンシブ切替 → Layout変更 → Reflow → Layout Shift → Scroll Anchoring補正 → 体感ズレ`

この鎖で起きることが多い。

## 画像が絡む時系列（重要）

画像は非同期ロードなので、次の順になりやすい。

1. 初期 `Layout` / `Paint`
2. JSでスクロール補正
3. 画像ロード完了
4. 画像サイズ確定で再 `Layout` / 再 `Paint`
5. 補正後の位置がまたずれる

つまり「JSが遅い」のではなく、
**JSの後に再レイアウト要因（画像確定）が来る**のが本質。

## スクロール時の見え方

- 単純スクロールは主に再描画（repaint）
- 高さが変わるイベントが混ざると再レイアウト（reflow）も発生
- 画像未予約だと、この再レイアウトが遅れて発火しやすい

## 防止策で何が変わるか

- `width` / `height` 属性: HTML段階で比率予約（最速）
- `aspect-ratio`: CSS適用段階で比率予約（実用上ほぼ同等）
- 予約後は、画像ロード時の高さ変化が減り、JS補正が壊れにくくなる

## 横幅変更が縦ズレを生む理由

- `max-width` や `padding` の変更は横方向の操作でも、テキスト折り返しを変える
- 折り返し行数が増減すると要素高さが変わる
- 結果として section 高さとスクロール位置がズレる

つまり「横幅の変更は直接ではなく、改行を介して縦方向に波及する」。

## CSS側で先に効く3点

1. `width` / `height` または `aspect-ratio` で画像高さを予約する
2. アンカーずれがあるなら `scroll-margin-top` を入れる
3. 画面外描画が重い場合は `content-visibility: auto` と `contain-intrinsic-size` を検討する

```css
main section[id] {
  scroll-margin-top: 80px;
}

main section {
  content-visibility: auto;
  contain-intrinsic-size: 800px;
}
```

補足:
- `content-visibility` は万能ではないため、要素特性やブラウザ差分を見ながら限定適用する。

## カンプのレイヤー寸法をCSSの箱へ直訳しない

デザインツールで表示されるテキストレイヤーの `H` は、背景を含む親要素の高さと同じとは限らない。

CSSへ移すときは次を分けて確認する。

1. `font-size`
2. `line-height`
3. フォントの字面とメトリクス
4. 親要素の `padding`
5. 背景・罫線を持つ要素の実寸

```css
.notice-band {
  font-size: 14px;
  line-height: 1.6;
  padding-block: 3px;
  background-color: #006ea8;
}
```

- テキストレイヤーが21pxでも、背景帯全体が21pxとは限らない。
- まず行の高さと親の内側余白から組み立てる。
- `height` 固定は、部品全体の高さが要件として確定している場合に使う。
- `margin-top: -1px` などの微調整は、通常配置・境界線・端数丸めを確認した後の最終手段にする。

## 対処順

1. 画像/動画のサイズ予約（`width`/`height`, `aspect-ratio`）
2. 高さが後から変わる要素を減らす（`fit-content` などを点検）
3. それでもズレる箇所に限定して `overflow-anchor: none;`
4. 最後に JS 補正を検討（CSS安定化後）

## 例

```css
.portfolio__list {
  overflow-anchor: none;
}
```

- 乱用せず、問題箇所だけに当てる。

## DevToolsでの見方（Computed / Overlay）

### Computedの値

- Computedは「最終採用された結果寄りの値」を見る場所
- 薄い灰色の `height: 439px` などは、CSSに直書きした値ではなく
  レイアウト計算の結果として表示されることがある

### Grid Overlay

- 紫の斜線は `gap` の可視化
- `gap` 数値はその場で直接表示されないことがある
- 空きトラック幅は `Show track sizes` で確認する

### Flexの `space-between`

- Gridのように「この空きが何px」と直接出しにくい
- 実務では `getBoundingClientRect()` で要素間を測る方が速い

## `@media` で「効いていないはずが効く」時の確認順

- 症状:
  - PC用に書いたつもりの `display: flex` がSPでも効く
- 主因:
  - `@media` の `}` が早く閉じて、意図したルールがブロック外へ漏れている
- 確認順:
1. 対象要素を親コンテナで選ぶ（`li` ではなく `ul` など）
2. `Computed` で該当プロパティ（例: `display`）を確認する
3. 値の出どころ（ファイル/行）で `@media` 内外を確認する
4. `Styles` で `}` の閉じ位置を見直す

- 補足:
  - 「要素サイズが 529px だからSP判定」という読み方は誤り。`min-width` 判定は viewport 幅で行われる。

## 新しめ機能の導入判断（CSSネスト / Sass）

- CSSネストはモダン環境で実用化が進んでおり、「ネスト目的だけ」ならネイティブCSSでも運用できる。
- ただし次は別判断が必要。
  - 古め端末を広く含む案件
  - build前提の設計（mixin/関数/分割統合）
  - Sassの `&__element` 連結前提の命名運用

注意点:

- ネイティブCSSネストでは Sass と同じ連結記法（`&__child`）をそのまま使えない。
- 導入判断は「書けるか」ではなく、対象ブラウザ要件と運用要件で決める。

判断順:

1. 対象ブラウザ要件を固定する
2. ネスト以外の機能要件（mixin/分割/連結命名）を洗い出す
3. ネイティブCSSで足りるか、Sass/PostCSSが必要かを決める
