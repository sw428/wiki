# 02_ケース_Blog一覧_SP基準設計と余白責務

## 記録日

- 5月4日

## このケースの前提

- 1つの貼り付けログに、構造判断（SP/PC）、BEM命名、余白設計、ボックスモデルが混在していた。
- そのため、最初に「話題を分解してから結論を戻す」認知手順を固定して検証した。

## 認知アプローチ（今回使った分解手順）

1. 話題を4層に分ける
- 構造層: HTMLを分けるべきか（意味 / 読み順）
- 命名層: BEM責務をどこに持つか
- 余白層: `gap` / `margin` / `padding` の責任分担
- 挙動層: `display` とボックスモデルでなぜその見え方になるか

2. SP基準で骨格を先に確定する
- SPで同じ流れに読めるなら、まず1一覧構造を維持
- PC差分はCSSへ寄せる

3. 「見た目差」と「意味差」を切り分ける
- 見た目差だけ: CSSで対応
- 意味差・操作差がある: HTML分離を許可

4. 距離の責務を固定して検証する
- 同種反復（カード同士）: `gap`
- 別ブロック間: `margin-bottom`（現運用に整合）
- 線と中身: `padding-top`

## 話題別の結果

### 1. 構造判断（1グリッドか分離か）

- 1グリッドで成立する条件:
  - SPで同一一覧として読める
  - PC差分がサイズ差・線などの見た目差に留まる
- 分離が有効な条件:
  - `featured` と `list` が意味として別グループ
  - 線の上下距離を厳密に制御したい

### 2. 命名判断（BEM）

- 親ブロックが `blog-articles` なら要素名は `blog-articles__...` に統一する。
- `blog__more` のような途中切替は、責務追跡を難しくするため不採用。

### 3. 余白判断（gap/margin/padding）

- `gap` と子要素 `margin` を同時に使うと距離責務が重なる。
- 本ケースでは、一覧間隔を親の `gap`、次ブロック距離を `margin-bottom` に分離した方が再利用しやすい。

### 4. 挙動理解（ボックスモデル）

- `margin` は外側距離なので、`border` も一緒に動く。
- `padding` は内側距離なので、`border` と中身の間を作る。
- 線の上下距離を説明するときは、`margin-top`（線の外）と `padding-top`（線の内）を分けると混線しない。

## どういうことが分かったか

- 1ファイルに話題が混在していても、層ごとに分解すると結論が安定する。
- 「SP基準で骨格固定 -> PC差分をCSSで吸収」の順は、学習時の認知負荷を下げる。
- BEM命名と余白責務は同時に決めると崩れにくい（命名だけ先行だと破綻しやすい）。
- 例外を消すより、例外条件を短く残した方が次のケース判断が速くなる。

## 採用した最小実装テンプレ

```scss
.blog-articles__list {
  display: grid;
  row-gap: 40px;
  margin-bottom: 64px;
}

.blog-articles__more {
  text-align: center;
}

@media (min-width: 768px) {
  .blog-articles__list {
    grid-template-columns: repeat(6, 1fr);
    column-gap: 30px;
    row-gap: 52px;
  }

  .blog-card {
    grid-column: span 2;
  }

  .blog-card--large {
    grid-column: span 3;
  }
}
```
