## 運用メモ（合意）

- ユーザーが見る導線改善を優先し、Codex 側都合だけの装飾やリンクは増やさない
- 既出内容の重複説明は避け、必要最小限で更新する
- 会話で確定した運用は、次回も使える形でこの repo に意図が分かる粒度で残す
- 通常は自律実行し、共有は「判断が割れる点」と「後で面倒化しやすいリスク」に絞る
- システムファイル（`README.md` / `CURRENT.md` / `AGENTS.md`）の変更時のみ、事前確認を行う
- それ以外の内容ファイル（各カテゴリ配下の md）は、Codex が確認なしで反映してよい（ユーザーが反映後に再確認する前提）
- フロントエンド知識に関係する内容を優先し、無関係なプライベートの個人話題は記録しない
- 1階層目（top直下）のフォルダは基盤として固定し、新規作成 / rename / move / delete は明示指示がある時だけ行う
- 2階層目以降のファイルカテゴリは、既存追記で不足する場合に限って Codex が必要分を追加してよい
- 2階層目以降は、導線改善や重複解消のために Codex が rename / move を行ってよい（判断理由を残す）
- `01_基礎概念` 配下は、分かりやすさ優先でカテゴリ追加を許容し、無理に既存カテゴリへ収めない
- `01_基礎概念` 配下は、必要に応じてファイル数の増加も許容する（可読性・再検索性を優先）
- この方針は基礎概念内の整理方針であり、1階層目固定基盤ルールとは分けて扱う
- delete は階層を問わず、明示指示がある時だけ行う
- 最短経路より、わかりやすさと認知負荷の低さを優先する（必要なら遠回りで説明する）
- 一般化を急がず、対象層と観測視点を先に固定してから判断ログ（条件 / 理由 / 例外）を残す
- 実践ケースは `02_実践/ケース検証` で管理し、完了後に再利用できる原理だけを `01_基礎概念` へ戻す
- 規約・命名・構造・素材・納品条件の話で案件名が不明な場合は、反映前にどの案件か確認する
- 公開・記録範囲はポートフォリオ全体の共通方針へ置き、実践側には指示の詳細ではなく自分の判断・工夫を残す
- 案件先の要望や既存の型は、教材内容を復元できない粒度で扱い、普遍ノートへ自動一般化しない
- 案件名・素材・固有数値を外しても成立する原理は、基礎概念の既存ノートを確認してから統合する
- 案件ファイル末尾に「基礎概念へ戻した内容」を置き、反映先と一般化した要点を追跡できるようにする
- 学習寄りの単独検証は `01_基礎概念/ケース検証` で扱う
- 同一回で複数議題が混在する場合は、1ページ内で `ケース1 / ケース2 / ...` に分割して受ける
- ケース記録は要約しすぎず、途中判断が追える丁寧な解説を優先する（多少雑多でも可）
- ケースから再利用できる原理・判断軸だけを基礎概念へ統合し、案件固有の条件・制約・最終判断は実践側に残す
- ケース本文の微修正は、Codex が再利用価値ありと判断した時点で基礎概念へ自律同期してよい（都度確認不要）
- ユーザーは問題提起・検証、Codex は解析・要約・統合・可読化を担当する分担で運用する
- 要点整理後はユーザー/GPT/Codex の三者で都度確認し、判断軸と要約スタイルを段階的に擦り合わせる
- 判断軸やまとめ方は急変させず、小さな差分で連続改善する
- 長文ログ圧縮・論点抽出・統合反映など処理負荷の高い整理は Codex 主担当で進める
- 反映先カテゴリが不足している場合は、2階層目以降で新規カテゴリを追加してよい
- ケース反映の要約はケースファイル末尾の「基礎概念への反映先」に集約する
