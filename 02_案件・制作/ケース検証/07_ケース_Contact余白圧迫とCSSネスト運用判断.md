# 07_ケース_Contact余白圧迫とCSSネスト運用判断

## 1. 事実（案件文脈）

- 対象は `contact-page` 実装時のCSS相談ログ。
- 同一回で次の議題が混在した。
  - `contact-info` の右側が詰まる（`flex` + `padding` + `clamp()` の挙動）
  - `padding-left` の `clamp()` と `padding-right` の優先順位誤解
  - `p/h` と `min-width: auto` の関係、`flex/grid` 文脈での縮み拒否
  - CSSネストが使える時代にSassが不要かどうか
  - `subgrid` の使いどころ
- 受領ログには業務メール文面の相談も含まれていたが、フロント実装スコープ外として本ケースでは扱わない。

## 2. 判断プロセス

### ケース1: `contact-info` の右側圧迫の主因

#### 候補A: `clamp()` が後に書かれているので優先され、右paddingが効かない

- 判定: 不採用
- 理由: `padding-left` と `padding-right` は別プロパティで競合しない。カスケード順の問題ではない。

#### 候補B: `flex-basis` と左右paddingの合算で内容幅が圧迫される

- 判定: 採用（既定）
- 理由: `flex: 1 1 391px` はベースサイズ計算に参加し、そこに左右paddingが加わる。横幅予算が厳しい時に内容領域が先に圧迫される。`box-sizing: border-box` でも、flexの横幅配分では `flex-basis` / `padding` / `gap` / 親幅 の予算合算次第で内容領域が想定より狭くなる。

#### 候補C: テキスト自体に `min-width` があるため必ず押し広がる

- 判定: 不採用
- 理由: テキストノード自体に `min-width` プロパティはない。縮み拒否は主に「flex/gridアイテムの自動最小サイズ（`auto`）」として現れる。

### ケース2: `min-width: auto` の理解

#### 候補A: `p/h` はデフォルト `min-width: 0`

- 判定: 不採用
- 理由: `min-width` の初期値は `auto`。ただし通常フローのブロック文脈では、自動最小サイズが問題化しにくく、実務上は親幅に従って折り返すように見える場面が多い。

#### 候補B: `flex/grid` の子では `auto` が最小内容幅（min-content）として効きやすい

- 判定: 採用
- 理由: 長い単語やアイコン+テキスト構成で縮みにくくなり、はみ出しや圧迫が起きやすい。必要に応じて `min-width: 0` を明示する。

### ケース3: CSSネスト時代のSass採用判断

#### 候補A: ネストが使えるのでSassは不要

- 判定: 条件付き採用
- 理由: ネスト目的だけならネイティブCSSで成立する。

#### 候補B: Sassはネスト以外の機能要件で判断する

- 判定: 採用（既定）
- 理由: mixin/関数/分割運用や広い互換性要求があるならSass（またはPostCSS）継続の価値がある。

#### 候補C: `subgrid` は即時必須

- 判定: 不採用
- 理由: まず通常のGrid/Flexで責務分離を固め、親子で同じグリッド線を共有したい段階で導入するのが再現性が高い。

## 3. 案件固有として残すもの（02側）

- `contact-info` は `@media (min-width: 768px)` で `display: flex` へ切替わるため、SPで見えない圧迫がPC幅で表面化しやすい。
- `padding-left: clamp(...)` は「左paddingの値決定」であり、右paddingとの優先関係はない。
- まずは次の順で修正すると再現しやすい。
  1. `box-sizing` を確認（プロジェクト既定が `border-box` か）
  2. `flex-basis` を見直す（例: `391px` 固定から `0` ベースへ）
  3. `min-width: 0` をアイテム/必要な子要素へ付与
  4. 長文には `overflow-wrap` を追加

## 4. 本線へ戻すもの（01側）

- 再利用可能な原理:
  - カスケード順は「同一プロパティ競合」で効く。左右paddingのような別プロパティには当てはめない。
  - `min-width: auto` は文脈依存で見え方が変わる（通常フローとflex/gridで挙動差が出る）。
  - テキストノードはCSSボックスではない。縮み拒否は親アイテム側の最小サイズ規則で説明する。
  - CSSネスト導入可否は「書けるか」ではなく「互換要件と運用機能」で決める。
  - `subgrid` は「親のトラック線共有」が必要な時に使う。
- 判断順:
  1. 圧迫の発生文脈を固定（通常フローかflex/gridか）
  2. 幅予算（`flex-basis` / `width` / `padding`）を分解
  3. 自動最小サイズの影響（`min-width: auto`）を確認
  4. `min-width: 0` と改行制御で縮み許可を与える
  5. 機能導入（ネスト/subgrid）は互換要件と保守要件で判定
- 例外条件:
  - 可読性上、行分割を許容できないラベルは `nowrap` を残し、親幅や列設計側で吸収する。
