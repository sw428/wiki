# マリンプラクティス 実装で意識したこと

## このファイルの位置づけ

- このファイルは、マリンプラクティスに取り組む中で自分が意識したこと・判断理由・工夫を残す。
- 公開・記録範囲は、親フォルダの [ポートフォリオサイト 公開・記録方針](../00_公開・記録方針.md) に従う。
- 課題や指示の内容ではなく、自分がどう考えて実装したかを中心に書く。

## 既存の構造を尊重する

- 最初から自分の命名へ置き換えず、既存HTML・既存クラスが担っている役割を先に確認した。
- `.container` や見出し用クラスなど、すでにある名前を活かしながらCSSを組み立てることを意識した。
- `l-` / `t-` 系の命名を使うこと自体を目的にせず、レイアウト・タイポグラフィ・コンポーネントの責務を分けて考えた。
- クラス名を変えられない場合でも、設計の考え方は既存クラスへ割り当てられると整理した。

```css
/* layout */
.container {
  max-width: 1080px;
  margin-inline: auto;
  padding-inline: 16px;
}

/* typography */
.heading-ttl {
  font-size: 22px;
  font-weight: 700;
  line-height: 32px;
}
```

## 作成済みのタイポグラフィを無駄にしない

- 先に考えた `t-` 系クラスをそのままHTMLへ足せない場合も、文字サイズ・太さ・行間の設計値は既存クラスへ移せると考えた。
- 使用されないクラスを残すのではなく、実際に使うセレクタへ整理し直すことを意識した。
- 同じ値を複数箇所で使う場合は、CSSカスタムプロパティやSassの変数・mixinへまとめる方向で考えた。
- 命名よりも「どの文字役割を担当するか」を追える状態を優先した。

## 詳細度を必要以上に上げない

- 場所ごとの差分を付けるときも、最初からIDを含む強いセレクタに頼らないようにした。
- 既存クラスで区別できるか、Modifier相当の考え方を使えるかを先に検討した。
- 強いセレクタが必要な場合は、なぜ必要なのかと影響範囲を意識するようにした。

## 画像素材をそのまま受け入れず整理する

- デザインツール上で別レイヤー・別オブジェクトなら、見た目が同じでも別ファイルとして書き出されることがある。
- 同じ見た目のSVGが複数あるときは、ファイル名だけで判断せず、形状・色・透過・内部参照の差を確認することを意識した。
- 描画差がなければ共通化できる可能性を考え、不要な素材重複を減らす視点を持った。
- 単色の長方形・円・線などは、画像のまま使うかCSSで再現するかを、変更しやすさと実装条件の両方から判断した。

## 曖昧な言葉をそのまま操作に変えない

- `sub` のように複数の意味を持つ言葉は、ブランチ・フォルダ・別リポジトリなど、何を指すかを先に分けて考えた。
- Git操作では「新しく追加する」と「現在のものを改名する」を別の目的として整理した。
- 公開設定も、pushしたブランチと実際の公開元が同じとは限らないため、設定側を確認する意識を持った。

## 今回得た整理

- 自分の設計を押し通すより、既存の構造を読み取って合わせる力も実務では重要。
- 命名規則は目的ではなく、責務を追いやすくするための手段。
- 作った設計案がそのまま使えなくても、考えた値や役割は既存構造へ再配置できる。
- 支給素材は全部をそのまま使うのではなく、差分・用途・管理方法を確認して整理する。
- 曖昧な依頼は、操作前に対象と目的を分けて確認する。

## 次回へ残す課題

- 実装開始前に、変更できる範囲と既存構造の依存先を確認する。
- タイポグラフィはクラス名だけでなく、役割と適用先の対応表まで作る。
- 素材整理では、参照状況と描画差を確認してから統合判断を行う。
- 強いセレクタや例外対応を使った場合は、理由を短く残す。
