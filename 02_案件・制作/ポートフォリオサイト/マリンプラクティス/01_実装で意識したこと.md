# マリンプラクティス 実装で意識したこと

## このファイルの位置づけ

- このファイルは、マリンプラクティスに取り組む中で自分が意識したこと・判断理由・工夫を残す。
- 公開・記録範囲は、親フォルダの [ポートフォリオサイト 公開・記録方針](../00_公開・記録方針.md) に従う。
- 課題や指示の内容ではなく、自分がどう考えて実装したかを中心に書く。
- 詳しい検証ログは [09_ケース_マリンプラクティス2カラムとCTA・Flex配置判断](../../ケース検証/09_ケース_マリンプラクティス2カラムとCTA・Flex配置判断.md) に分ける。

## 既存の構造を尊重する

- 最初から自分の命名へ置き換えず、既存HTML・既存クラスが担っている役割を先に確認した。
- `.container` や見出し用クラスなど、すでにある名前を活かしながらCSSを組み立てることを意識した。
- `l-` / `t-` 系の命名を使うこと自体を目的にせず、レイアウト・タイポグラフィ・コンポーネントの責務を分けて考えた。
- クラス名を変えられない場合でも、設計の考え方は既存クラスへ割り当てられると整理した。

```css
/* layout */
.container {
  max-width: 1080px;
  margin-inline: auto;
  padding-inline: 16px;
}

/* typography */
.heading-ttl {
  font-size: 22px;
  font-weight: 700;
  line-height: 32px;
}
```

## 作成済みのタイポグラフィを無駄にしない

- 先に考えた `t-` 系クラスをそのままHTMLへ足せない場合も、文字サイズ・太さ・行間の設計値は既存クラスへ移せると考えた。
- 使用されないクラスを残すのではなく、実際に使うセレクタへ整理し直すことを意識した。
- 同じ値を複数箇所で使う場合は、CSSカスタムプロパティやSassの変数・mixinへまとめる方向で考えた。
- 命名よりも「どの文字役割を担当するか」を追える状態を優先した。

## 詳細度を必要以上に上げない

- 場所ごとの差分を付けるときも、最初からIDを含む強いセレクタに頼らないようにした。
- 既存クラスで区別できるか、Modifier相当の考え方を使えるかを先に検討した。
- 強いセレクタが必要な場合は、なぜ必要なのかと影響範囲を意識するようにした。

## ブラウザ標準の挙動と自分の余白を分ける

- 要素が想定より広く見えたとき、すぐに `margin` や `padding` の問題と決めつけないようにした。
- 横幅では、ブロック要素が親幅まで広がる挙動と、実際に付けた余白を分けて確認した。
- 縦幅では、`line-height` が作る行の高さ、子要素の改行・積み重なり、`margin` / `padding` を別々に確認した。
- DevToolsでは、content・padding・border・marginの色分けと、子要素ごとの計算値を見て原因を切り分けた。

## リセットCSSの範囲を意識する

- `body { margin: 0; }` はページ外周の標準余白を消す指定であり、見出し・段落・リストなどの標準余白までは消さないと整理した。
- `* { margin: 0; }` で全要素を一括リセットする方法と、対象要素を列挙して必要な範囲だけリセットする方法の違いを意識した。
- リストでは `margin` だけでなく、標準の `padding` や `list-style` も別に確認した。
- リセットの短さだけでなく、「何を消しているか」が後から読める書き方を選ぶようにした。

## 画像の属性と表示上の比率を分ける

- HTMLの `width` / `height` 属性は、画像本来の比率と読み込み前の領域予約をブラウザへ伝える役割として考えた。
- 表示サイズや表示枠の比率はCSS側で調整し、属性値と見た目のサイズを同じ役割として混ぜないようにした。
- カンプに合わせて切り抜く場合も、画像そのものの比率、表示枠の比率、`object-fit` の役割を分けて判断した。
- エディタのショートカットで属性を簡単に入力できても、実画像の比率と一致しているかを確認してから使うようにした。

## 文字入り画像は比率枠へ押し込みすぎない

- 電話画像・予約画像・ロゴのように文字が切れると困る素材は、まず `width` と `height: auto` で自然に縮小する方を考えた。
- `aspect-ratio` は画像を小さく見せる指定ではなく、比率の箱を作る指定として分けた。
- 写真のように枠いっぱいへ見せたい場合は `object-fit: cover`、全体を見せたい場合は `object-fit: contain` と判断した。
- 文字入り素材で「もっと引いて見せたい」場合は、切り抜きよりも、そもそも比率維持表示で足りるかを先に確認した。

## 画像にも機械的にクラスを付けない

- BEMの形をそろえるためだけに、すべての画像へElementクラスを追加しないようにした。
- 画像自体へ幅・比率・切り抜き・角丸などを指定する場合は、画像用クラスを付けて責務を明確にした。
- 親要素のレイアウトだけで完結する場合や、共通の画像ルールで足りる場合は、クラスを増やさない選択肢も残した。
- クラス数ではなく、「その画像を個別に識別する必要があるか」で判断した。

## 命名はDOMの深さより役割から考える

- すべての `div` に機械的にクラスを付けず、CSS・JS・再利用・意味の識別に必要な要素へ絞ることを意識した。
- ページをヘッダー、ナビゲーション、主要セクション、サイド領域などの役割単位に分けてからBlock候補を考えた。
- `box` / `wrap` / `content` のような見た目だけの名前より、ニュース項目、操作領域、本文など役割が追える名前を優先した。
- BEMのElementはDOMの入れ子をそのまま写すのではなく、親Blockの中で担当する役割から決めた。
- 複数セクションで同じ見た目の見出しは、各BlockのElementへ重複させるか、共通タイポグラフィとして扱うかを再利用範囲で判断した。

## BEMを使うこと自体を目的にしない

- BEMを使わない場合も、セクションを示すクラスと繰り返し単位のクラスがあれば、必要な範囲へCSSを当てられると整理した。
- `.voice div div p:first-child` のようにDOM階層へ深く依存する指定は避け、構造変更の影響を受けやすい場所だけ役割クラスを付けるようにした。
- 既存の `.container` を `.l-container` へ変更する場合は、HTMLだけでなくCSS・JS・参照箇所を一式で確認する必要があると考えた。
- 命名体系を変えるかどうかは、見た目の好みではなく、案件の既存ルールと変更可能範囲を基準にした。

## 表示制御クラスを部品設計と混ぜない

- `.visible-sp` / `.visible-pc` は部品の形を作るクラスではなく、表示条件を追加する補助クラスとして扱った。
- `u-` 接頭辞を使うかどうかは案件の命名規則に合わせ、途中から自分の体系だけを混ぜないようにした。
- 表示・非表示の制御と、`flex` / `grid` によるレイアウト制御を別の責務として考えた。

## 外側の領域と内側のコンテンツ幅を分ける

- カンプの左右余白を、そのまま大きな固定 `padding` として写さないようにした。
- 外側は背景色・罫線・画面全体の領域、内側は `max-width` と中央寄せでコンテンツ位置を管理する形に分けた。
- 「全体幅 - 左右余白 = 中身幅」という見方から、固定余白ではなく内側コンテナの最大幅を考えるようにした。
- 線だけを広く伸ばしたい場合は、線を外側要素に持たせ、中身だけを内側コンテナへ収める責務分離を意識した。

## mainとasideは固定側・可変側を分ける

- 2カラムでは、親が横並びと列間隔を管理し、サイド領域は固定幅、メイン領域は残り幅を受ける形で考えた。
- Flexだから子要素に幅を書いてはいけない、ではなく、固定したい子だけ幅を持たせると整理した。
- サイド領域はバナーや画像が崩れやすいため `flex-shrink: 0` で潰さない対象にした。
- メイン領域は `flex: 1` と `min-width: 0` で、残り幅を受けつつ縮み拒否が出にくい形にした。
- 数値を直接写す前に、親幅・gap・固定サイド幅から、メインが結果として何px相当になるかを確認した。

## カンプのレイヤー寸法をCSSへ直訳しない

- カンプ上のテキストレイヤーの高さを、そのまま背景帯の `height` として使わないようにした。
- `font-size`、`line-height`、フォントの見え方、親要素の `padding` を分けて、帯全体の高さを組み立てた。
- 1pxの差が出ても、最初から負のマージンで合わせず、どの箱の寸法差なのかを確認してから微調整するようにした。
- 固定高はデザイン上の高さが明確な場合に限定し、通常は `line-height` と `padding` を基準にした。

## 入れ子のFlexは別々の整列範囲として考える

- 外側のFlexは、ロゴと操作ボタン群のような直下の子要素を整列するために使った。
- 内側の操作ボタン群もFlexにする場合は、その中のリンクや画像が別のFlex文脈へ参加すると整理した。
- 同じ `align-items: center` を二箇所へ書く場合も重複ではなく、それぞれ別の直下要素を中央にそろえる指定として考えた。
- 親のFlex指定が、孫要素の並び方まで自動的に整えるわけではない点を意識した。

## ハンバーガーボタンは外側の配置と内側の線を分ける

- SPヘッダーの青帯内でボタン自体を縦中央に置くのは、親の `align-items: center` の責務として考えた。
- `.hamburger-button` の `display: grid` は、ボタンの中の3本線を縦に並べるための指定として分けた。
- `place-content: center` は、ボタン箱の中で線を中央に置く指定であり、親帯の中でボタン自体を中央にする指定ではないと整理した。
- `width` / `height` を書かなくても線の大きさと `gap` で自然なボタンサイズはできるが、押せる領域を確保するため、ボタン側の箱サイズは明示する方針にした。

## SPヘッダーのメニューは通常フローを優先する

- ハンバーガーメニューを写真の上へ重ねたいのか、MVの前にSP用の行として置きたいのかを先に分けた。
- 今回は写真へ重ねる意図ではなく、SP用の青帯メニューとして置くため、`position` で無理に重ねず通常フローで管理する方を優先した。
- `visible-sp` / `visible-pc` は存在条件の切り替えとして使い、レイアウトや重なりの調整とは分けて考えた。
- SPでヘッダー全体に余白を入れたくない場合は、`site-header` 全体ではなく、余白が必要な内側要素へ `padding-inline` を分けるようにした。

## サイドバーの反復余白は最後に残る余白を意識する

- バナーや記事のような同種反復では、親で縦並びにできるなら `gap` を優先候補にした。
- 既存HTMLを大きく変えず、2個目以降にだけ距離を付けたい場合は、`.sidebar__banner + .sidebar__banner` のような隣接兄弟セレクタを使うと整理しやすいと考えた。
- 一律の `margin-bottom` は最後の要素にも効くため、次のブロックとの距離まで混ざらないか確認するようにした。
- 「同じ部品同士の間隔」と「次の別ブロックまでの距離」を別責務として見るようにした。

## バナー画像の上に文字を重ねる時は基準箱を作る

- サイドバーバナーの仮テキストは、画像の上に重ねる表示として扱った。
- 親の `.sidebar__banner` に `position: relative` を置き、テキスト側を `position: absolute; inset: 0;` で親枠いっぱいに広げる形にした。
- 文字の中央寄せは、絶対配置そのものではなく、テキスト箱の中で `grid` / `place-items` に任せると整理した。
- 親の高さは通常フローに残る画像が作り、絶対配置のテキストは親の高さを作らないと考えた。

## Voiceカードの右下配置は箱と文字を分ける

- お客様の声のメタ情報を右下へ置くとき、`.voice__meta` 自体を動かしたいのか、中の文字だけ右寄せしたいのかを分けた。
- 本文エリアを縦方向のFlexにし、最後のメタ情報へ `margin-top: auto` を使うことで、余った縦スペースを上側に吸収させた。
- 文字だけを右寄せする場合は `text-align: right`、要素の箱ごと右へ寄せる場合は `align-self: flex-end` と整理した。
- `.voice__meta { display: flex; }` だけでは、自分自身は右下へ動かない点を確認した。
- Flexは「その要素自身を動かす指定」ではなく、「直下の子要素の配置を管理する親にする指定」として扱った。

## ナビゲーションは文字位置だけでなく操作範囲で考える

- `text-align: center` は横方向の文字配置であり、縦中央やクリック範囲までは決めないと整理した。
- ナビゲーション項目では、`li` よりリンク要素側に高さ・padding・中央寄せを持たせ、クリックできる範囲を確保することを意識した。
- 通常は内容変化に耐えやすい `padding` と `min-height` を基準にし、均等幅がデザイン上の要件である場合だけ `width` を検討した。
- 区切り線、ナビ全体の上下余白、各リンクの操作範囲を別の要素へ分担させた。

## CTAボタンは文字だけかアイコン込みかでdisplayを選ぶ

- 文字だけの小さいリンクや文章中に自然に混ざるボタンは `inline-block` でも十分と整理した。
- 「一覧へ + 矢印」のように、文字とアイコンを上下中央にそろえたいCTAは `inline-flex` を優先した。
- ボタン面積はまず `padding` で作り、カンプ上の最低サイズを守りたい場合だけ `min-width` / `min-height` を足すようにした。
- 矢印が単純な文字なら `::after`、SVG画像として扱うなら `span + img` に分ける判断をした。
- `::after` はリンク要素のpadding内に入る疑似的な子要素として扱い、文字との間隔は `gap` か `margin-left` で管理した。

## 画像素材をそのまま受け入れず整理する

- デザインツール上で別レイヤー・別オブジェクトなら、見た目が同じでも別ファイルとして書き出されることがある。
- 同じ見た目のSVGが複数あるときは、ファイル名だけで判断せず、形状・色・透過・内部参照の差を確認することを意識した。
- 描画差がなければ共通化できる可能性を考え、不要な素材重複を減らす視点を持った。
- 単色の長方形・円・線などは、画像のまま使うかCSSで再現するかを、変更しやすさと実装条件の両方から判断した。

## 曖昧な言葉をそのまま操作に変えない

- `sub` のように複数の意味を持つ言葉は、ブランチ・フォルダ・別リポジトリなど、何を指すかを先に分けて考えた。
- Git操作では「新しく追加する」と「現在のものを改名する」を別の目的として整理した。
- 公開設定も、pushしたブランチと実際の公開元が同じとは限らないため、設定側を確認する意識を持った。

## 今回得た整理

- 自分の設計を押し通すより、既存の構造を読み取って合わせる力も実務では重要。
- 命名規則は目的ではなく、責務を追いやすくするための手段。
- 作った設計案がそのまま使えなくても、考えた値や役割は既存構造へ再配置できる。
- 見た目のズレは、標準挙動・ボックス・行・余白・親幅を分けると原因を追いやすい。
- 外側の装飾領域と内側のコンテンツ幅を分けると、固定値への依存を減らせる。
- 2カラムでは、固定したいサイド領域と可変にしたいメイン領域を分けると、Flexの幅配分を読みやすくできる。
- カンプのレイヤー寸法は、CSSの行・親ボックス・余白へ分解してから移す。
- 入れ子のFlexでは、各Flexコンテナが自分の直下要素だけを整列する。
- 右下配置では、要素の箱を動かすのか、中の文字だけを寄せるのかを先に分ける。
- ナビゲーションは見た目だけでなく、リンクのクリック範囲まで含めて設計する。
- CTAボタンは、文字だけなら `inline-block`、文字 + アイコンを内部整列するなら `inline-flex` が判断しやすい。
- 同種反復の間隔は、親で束ねられるなら `gap`、既存構造のまま2個目以降だけなら隣接兄弟の `margin-top`、別ブロックまでの距離なら `margin-bottom` と分ける。
- ハンバーガーは、親がボタンを置く責務と、ボタンが中の線を並べる責務を分ける。
- SP専用のメニュー行は、重ねる必要がなければ `position` ではなく通常フローで置く方が読みやすい。
- 文字入り画像は、切り抜き枠ではなく比率維持表示を先に考える。
- BEM・Utility・画像用クラスは、名前を増やすためではなく責務を識別するために使う。
- 支給素材は全部をそのまま使うのではなく、差分・用途・管理方法を確認して整理する。
- 曖昧な依頼は、操作前に対象と目的を分けて確認する。

## 次回へ残す課題

- 実装開始前に、変更できる範囲と既存構造の依存先を確認する。
- タイポグラフィはクラス名だけでなく、役割と適用先の対応表まで作る。
- 余白や高さの違和感が出たら、DevToolsでcontent・行の高さ・余白を順番に確認する。
- カンプの数値を移す前に、テキストレイヤー・親要素・背景帯のどの寸法か確認する。
- カンプの固定値をCSSへ移す前に、外側領域と内側最大幅のどちらを表す数値か確認する。
- 2カラムでは、固定側に `flex-shrink: 0`、可変側に `flex: 1` / `min-width: 0` が必要か確認する。
- Flexを入れ子にした場合は、各コンテナの直下要素と整列軸を図にして確認する。
- 右下配置や下寄せでは、`margin-top: auto` / `align-self` / `text-align` の対象差を確認する。
- ナビゲーションはリンク要素を基準に、操作範囲・中央寄せ・区切り線を分けて設計する。
- CTAの矢印は、疑似要素で足りるか、実画像として `span + img` に分けるかを先に決める。
- 反復要素の余白は、最後にも余白が残ってよいかを確認してから `gap` / 隣接兄弟 / `margin-bottom` を選ぶ。
- SPヘッダーでは、表示条件、通常フローでの位置、重ね表示を分けて確認する。
- 文字入り画像では、`aspect-ratio` で箱を作る前に、`width` + `height: auto` で足りるか確認する。
- 素材整理では、参照状況と描画差を確認してから統合判断を行う。
- 強いセレクタや例外対応を使った場合は、理由を短く残す。
